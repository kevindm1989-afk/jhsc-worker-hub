## Findings

### CRITICAL (release-blocker for M2.1)

_None._ The lifecycle ships with the snapshot copy guard, an env-driven signer-role architecture, mobile-primary input/touch-target hygiene that holds across every meeting form, and a tamper-evident chain that the security review already cleared. Findings below are S5-bundle work; the items in HIGH are mobile/a11y baseline regressions that should be addressed before the rep depends on the deployed build for a live meeting.

### HIGH (fix-in-S5-bundle)

- **F-P1: Print stylesheet does not expand collapsed `SectionAccordion` bodies — the printed minutes are partial whenever the rep prints from screen.**
  - **Where:** `apps/web/src/views/meeting-detail-view.tsx:689` (section body is conditionally rendered: `{expanded ? ( … ) : null}`); `apps/web/src/index.css:258-263` (the print block explicitly DECLINES to force-expand: comment says "A future S4/S5 enhancement could expose a 'Print all sections expanded' toggle … the spec accepts the on-screen collapse state.").
  - **What:** A rep tapping print after a meeting sees only the section they last expanded. Every other section ships to paper with title + chips but no notes, no attendance roll-up, no quorum chip. The CSS comment self-documents the gap.
  - **Why it matters:** CLAUDE.md "Print stylesheet for every printable view — evidence-grade output" + ADR-0012 §3.5 ("the printable layout is the canonical minutes printout … the rep's preview-before-export"). If the rep prints the minutes for a binder ahead of the 2.3 server-rendered PDF being available, the print is missing the operational record. Worker-side evidentiary defensibility takes the hit.
  - **Fix:** Either (a) render every accordion body unconditionally and rely on screen-mode `[aria-expanded="false"] + div { display: none }` so the print stylesheet flips that off, or (b) on `window.matchMedia('print').addEventListener('change', … )` (or `beforeprint`), set `expandedSectionId = 'all'` and render every body. Pattern (a) is simpler and matches how `<details>` is handled in the same stylesheet (line 219-228).

- **F-P2: Recommendations + roll-call-quorum + meetings-list copy cites OHSA / CLC by string only — no `<CitationRef />` hover surface.**
  - **Where:** `apps/web/src/views/minutes-view.tsx:82` ("21-day s.9(21) clock"), `:309` ("OHSA s.9 / CLC s.135"); `apps/web/src/components/meetings/section-notes-sheet.tsx:194` (the `recommendations` section title is the bare string "Recommendations"); `apps/web/src/views/meeting-detail-view.tsx` (no statute reference on the recommendations accordion header); `apps/web/src/meetings/rights-protective-copy.ts:52` ("s.50 (OHSA) or s.147 (CLC Part II)" rendered as raw text in `staleManagementSignatureHint`).
  - **What:** CLAUDE.md "Citation Hover" is listed as one of three signature interactions: "Hover (desktop) or tap-and-hold (mobile) any OHSA/regulation reference anywhere in the app". The QuorumChip wires `<CitationRef statute={…} citation={…} />` correctly — every other statute mention in the M2.1 surface is a bare string. The rep cannot tap-and-hold "s.9(21)" on `/minutes` to confirm the corpus version. The 30-day stale-management hint surfaces s.50 / s.147 reprisal pathways as plain text — the rep who is potentially exercising a reprisal claim has the highest legitimate need to see the corpus clause before deciding.
  - **Why it matters:** Non-negotiable #5 (legal citations come from the corpus) is structurally fine — the strings are accurate — but the user-facing signature interaction is the trust mechanism. A statute reference that doesn't expand is indistinguishable in the rep's mind from fictional citation. Mobile-primary impact: the rep on a phone in a meeting cannot quickly confirm "what does s.9(21) actually say" without leaving the view.
  - **Fix:** Wrap every OHSA / CLC reference in the M2.1 surface with `<CitationRef />`. Inventory: `minutes-view.tsx:82,309`; `humaniseSectionType` (add `recommendations → "Recommendations · OHSA s.9(20) / CLC s.135(5)"` with the citation half wrapped); the recommendations accordion header in `meeting-detail-view.tsx`; the `staleManagementSignatureHint` text (split into prose + two `<CitationRef />` instances). The standing-agenda anchor (OHSA s.9(18-19)) should also surface on the agenda template chip near `new-meeting-view.tsx:269` ("agenda v1 … standard JHSC agenda template").

- **F-P3: Pull-to-refresh missing on `/minutes` — non-negotiable #9 ("Pull-to-refresh with haptic feedback") regression.**
  - **Where:** `apps/web/src/views/minutes-view.tsx:167-211` (`MeetingListInner` uses a one-shot `useEffect` fetch with no refresh affordance). No PullToRefresh primitive exists in the codebase — only a touchstart→touchmove comment in `sync/components/sync-panel.tsx:337`.
  - **What:** The Minutes tab is the canonical landing route per non-negotiable "Minutes-centric"; a rep returning to the tab after a sync drain on another device has no way to pull the latest list without navigating away and back. CLAUDE.md Mobile-Primary Patterns lists pull-to-refresh as a baseline.
  - **Why it matters:** Less load-bearing than F-P1/F-P2 but is a known-regression against the documented mobile baseline. The 2.1 milestone is the first time the Minutes tab carries real data; the pattern needs to land before the rep depends on the live build.
  - **Fix:** Build a `PullToRefresh` wrapper (touchstart/touchmove/touchend → translate Y + spinner → release triggers a refetch). Wire it around `MeetingListInner`. Add the same wrapper to `meeting-detail-view.tsx`'s body for in-meeting state pulls. Use Vibration API for the threshold-crossed haptic per the design rule.

### MEDIUM

- **F-P4: Attendance + section-notes sheets have no Escape-key handler, no focus trap, and no focus return on close.**
  - **Where:** `apps/web/src/components/meetings/attendance-sheet.tsx:150-307` and `apps/web/src/components/meetings/section-notes-sheet.tsx:85-173`. Both render `role="dialog" aria-modal="true"` but neither binds a `keydown` listener for Escape, neither traps Tab inside the sheet, and neither restores focus to the trigger button after close. The attendance sheet also does not autoFocus the first input (the section-notes sheet does autoFocus the textarea — `:138`).
  - **Why it matters:** WCAG 2.2 AA dialog conformance — keyboard users tab past the close button into the page behind, and Escape (the universal modal dismissal key) does nothing. ADR-0012 §3.5 explicitly calls out "Escape collapses" as a keyboard contract. The sheets are the most-used mobile affordances on the live meeting view; a screen-reader user roll-calling attendance has the worst experience of any 2.1 flow.
  - **Fix:** Extract a `<BottomSheet>` primitive in `apps/web/src/components/ui/` that handles (a) `onKeyDown` Escape → onClose, (b) focus trap via `react-focus-lock` or a manual tab cycle, (c) `useEffect` cleanup that calls `previouslyFocused.current?.focus()`. Use it in both sheets + audit the recommendation-response-sheet / inspection-signature-sheet for the same gap.

- **F-P5: QuorumChip has no `aria-live` — screen-reader users miss quorum transitions.**
  - **Where:** `apps/web/src/components/meetings/quorum-chip.tsx:32-44`. The chip carries `aria-label` (good) but is rendered in a static `<div>`; when attendance toggles flip `compliant: false → true`, the change is silent for AT users. The stale-management-hint at `meeting-finalization-view.tsx:257-270` correctly uses `role="status" aria-live="polite"`.
  - **Why it matters:** Quorum is the load-bearing operational threshold for the meeting; missing the announcement that quorum landed (or dropped after an early-departure flip) is a real workflow gap for a rep on a screen reader. ADR-0012 §3.5 lists "Screen-reader announcements on section advance" as a baseline; quorum is at the same tier.
  - **Fix:** Wrap the QuorumChip's outer `<div>` in a polite live region: `<div role="status" aria-live="polite" aria-atomic="true">…</div>` and ensure the `aria-label` text changes when compliance flips.

- **F-P6: 0-attendee empty state on roll-call section is silent — the rep gets no nudge toward attendance capture.**
  - **Where:** `apps/web/src/views/meeting-detail-view.tsx:383-385` (`detail.attendance.length === 0 → "No attendees recorded yet."`). The roll-call section accordion (`:691-701`) renders the QuorumChip when expanded but does not surface a "Tap '+ Add' to capture the roster" affordance when attendance is empty.
  - **Why it matters:** CLAUDE.md "Empty states do work. Never 'No data.' Show what to do next." The current copy is exactly the rule's anti-pattern. Quorum threshold cannot compute without members; the rep on a 390px screen who lands on the roll-call accordion with zero attendees gets no instruction.
  - **Fix:** When `attendance.length === 0`, the aside attendance card renders "No attendees recorded yet — tap Add to capture each member as they call out." The roll-call section body, when expanded, renders an inline pointer to the same affordance. Mirror the pattern from `apps/web/src/views/minutes-view.tsx:293-321` (empty state with statute anchor + CTA).

- **F-P7: `data-print="evidentiary"` evidentiary footer on meeting-detail uses `hidden … print:block` instead of being driven by the `[data-print='evidentiary']` rule in `index.css`.**
  - **Where:** `apps/web/src/views/meeting-detail-view.tsx:467` — `className="mx-4 mt-6 hidden text-xs md:mx-6 print:block"` is the only place in the 2.1 surface that uses the inline `print:block` utility instead of trusting the `data-print="evidentiary"` rule (`apps/web/src/index.css:266-274`).
  - **Why it matters:** Inconsistent application of the canonical convention documented in CLAUDE.md ("Print attribute convention. … data-print='evidentiary' … the canonical way to author a new printable view"). The block does carry the attribute, so it would already surface via the rule — the `hidden print:block` is redundant and breaks the "single source of truth" pattern. A future refactor that touches the global rule will not also flip this view.
  - **Fix:** Drop the `hidden print:block` from the className; the `data-print="evidentiary"` rule already provides `display: block !important` on print. Default the element to screen-hidden via the rule rather than the inline utility (the rule does not yet hide it on screen — but the existing visual posture is "muted on screen, surfaces on print," so add a screen-mode `[data-print="evidentiary"] { display: none }` paired with the print-mode block override). This is the load-bearing pattern for evidentiary metadata across the app.

### LOW

- **F-P8: `humaniseSectionType` omits `complaints_review` from the section accordion title set on print.**
  - **Where:** `apps/web/src/components/meetings/section-notes-sheet.tsx:176-203` does map `complaints_review → "Complaints review"`, but the v1 template seed (`apps/api/scripts/seed-meeting-template.ts:106-172`) deliberately omits `complaints_review`. The function correctly handles it as a forward seam. No bug, but a defensive test asserting the function covers every enum value would catch a future enum addition.
  - **Fix:** Add a vitest case that iterates `meetingSectionType` and asserts each value produces a non-empty string from `humaniseSectionType`.

- **F-P9: Section accordion's expand/collapse affordance has no visible icon — only the border color changes.**
  - **Where:** `apps/web/src/views/meeting-detail-view.tsx:642-687` — the accordion button shows running/done badges but has no chevron-down → chevron-up rotation on toggle. Visually subtle; the design rule "Always pair color with icon or label — never color alone" applies.
  - **Fix:** Add a `<ChevronDown />` at the right of the accordion button that rotates 180° when `expanded === true`. Color-blind users currently distinguish expand state by border color alone (`border-primary/40` vs `border-border`).

- **F-P10: `MEETING_STATUS_LABELS` is exported from a default-named alias at the bottom of `minutes-view.tsx` and re-imported by `meeting-detail-view.tsx`.**
  - **Where:** `apps/web/src/views/minutes-view.tsx:323` (`export { STATUS_LABELS as MEETING_STATUS_LABELS }`) → `apps/web/src/views/meeting-detail-view.tsx:55` (`import { MEETING_STATUS_LABELS } from './minutes-view'`).
  - **Why it matters:** Cross-view import couples the detail view to the list view's internal labels; if `minutes-view.tsx` is later split or the labels move to `shared-types`, the detail view breaks. This is a code-organization nit, not a privacy or rights-protective concern.
  - **Fix:** Move the constant to `apps/web/src/meetings/labels.ts` (new file) alongside `ATTENDANCE_ROLE_LABELS` and the status-icon/color maps. Both views import from there.

- **F-P11: `MeetingFinalizationView` ships `<input … className="bg-card …">` while `NewMeetingView` ships `<input … className="bg-background …">` for visually identical fields.**
  - **Where:** `apps/web/src/views/meeting-finalization-view.tsx:530,688,709,750` vs `apps/web/src/views/new-meeting-view.tsx:207,225,246,262`. Both views render fields on a `bg-card` parent, so the finalization fields blend into the parent while the new-meeting fields contrast.
  - **Fix:** Standardize on `bg-background` (the new-meeting pattern) for input contrast, or extract a shared `<TextField />` primitive.
