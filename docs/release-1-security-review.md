## Findings

### CRITICAL (release-blocker)

- **F-S1: Fuzz `failures/` dump directory is not gitignored.**
  - **Where:** `packages/excel-import/test/fuzz/failures/` — directory exists in-tree (created by `mkdirSync` in `beforeAll`, `fuzz.test.ts:101`). Repo root `.gitignore` does not list it (verified by `git check-ignore` returning no output and no `packages/excel-import/.gitignore` file).
  - **What:** `dumpFailure()` in `fuzz.test.ts:115-140` writes up to 4 KB of raw adversarial bytes plus a `.meta.txt` carrying the seed, scenario, hex preview, and uncaught-throw message for every failing case. The directory is real on disk after a single CI run and any future failure would land artifacts that a routine `git add packages/excel-import/test/fuzz/` would sweep into a commit. Combined with the fact that some scenarios embed prototype-pollution payloads + bidi spoofing strings + corrupted-xlsx bytes, this is a vector for accidentally committing adversarial blobs and (more concerning) for committing whatever real workbook bytes a developer might have fed into a local re-run of the harness.
  - **Why it matters:** CLAUDE.md non-negotiable #4 (privacy-by-default, minimize collection). SECURITY §2.12 T-HD18 frames the harness as the structural backstop on the 1.11 attack surface; the harness itself becoming an exfiltration channel for failure samples (synthetic OR real, if a developer re-runs locally against a real workbook) is a regression on that posture. The S4 brief item this was meant to catch.
  - **Fix:** Add `packages/excel-import/test/fuzz/failures/` (or `**/test/fuzz/failures/`) to the repo `.gitignore`. Also remove the empty `failures/` directory currently committed (replace with a `.gitkeep` outside the ignore pattern or simply do not pre-create it; let the `mkdirSync` create it at run time only). Add a CI lint that fails the build if any file under that path is staged. Recommended belt-and-suspenders: have `dumpFailure` redact / truncate the meta file to scenario + seed + index ONLY (drop the hex preview and the raw bytes) when an env var like `FUZZ_DUMP_BYTES=1` is unset — the seed alone reproduces the case.

### HIGH (fix-in-S5-bundle)

- **F-S2: `verifyChainFull` reports `ok: true` when window is unbounded but rows are non-empty and start at idx > 0 — but the sentinel runs AFTER the loop already mutated `prevIdx`, so the in-loop "idx must equal prevIdx+1" check on a non-zero starting row is silenced too.**
  - **Where:** `apps/api/src/lib/audit-verify-full.ts:158-296`. On the first iteration (`i === 0`), the code only checks `prevIdx !== null` (line 167) — it is null on the first row, so any first-row idx (including 100, 1000, or 1) skips the gap detector. The post-loop sentinel at lines 290-296 is the only place this gets caught for an unbounded window. But the sentinel only fires when `rows.length > 0`. Acceptable, but the divergence list will then carry a single `idx_gap` event whose `previousIdx: null` is structurally indistinguishable from "this was an intentional windowed walk that started at idx>0" — except the windowed-walk path takes a different branch (`window.sinceIso !== null`). Functionally correct, but the JSON report's consumer cannot tell whether a `previousIdx: null` gap means "the chain doesn't start at genesis" (release-blocker) versus "deliberately windowed." The runbook (`release-1-backup-restore-runbook.md` §2.3) just says "`gaps: []`" — operators may not realize a leading-gap divergence even appears in their report.
  - **What:** Two related issues:
    1. The check is correct but the **JSON divergence shape elides the distinguishing context**. A consumer reading `divergences[0]={kind:'idx_gap', idx:5, previousIdx:null}` cannot tell whether `window.sinceIso` was set without cross-referencing the `window` field. The runbook should explicitly tell the operator to verify the `window.sinceIso` is null AND the leading-gap divergence appears together.
    2. The "actor_timestamp_regression" check has a subtle race condition: per-actor map is updated **after** the divergence is pushed (lines 268-279 inside the same per-row block — fine), but the sentinel logic in 286-296 runs only for `window.sinceIso === null && window.fromIdx === null`. If a future caller sets `fromIdx` but leaves `sinceIso` null (or vice versa), the sentinel's "leading gap" detection is suppressed silently.
  - **Why it matters:** Non-negotiable #2 (tamper-evident audit chain). T-HD24 explicitly promises "the structured report's `gaps` array vs `hashMismatches` array is the structural disambiguation." A leading gap is a real tamper signal (someone deleted the genesis row); silencing it via window-flag inconsistency would be a procedural backdoor.
  - **Fix:** (a) Replace the post-loop sentinel with a richer first-row check inside the loop body (when `i === 0 && r.idx !== 0 && window` indicates unbounded, emit the divergence with `previousIdx: null` and a `detail` string that names which window flag was checked). (b) Add a `windowApplied: boolean` to the divergence shape so the JSON consumer can disambiguate. (c) Add an explicit pre-loop sanity assertion that `(sinceIso === null) === (fromIdx === null)` (or document that they must move together) — otherwise the windowed branch logic in 290-296 is brittle to future flag additions.

- **F-S3: Mobile section-move test bypasses the UI entirely; it does not verify the production audit-event payload — it verifies that the test fixture's POST body matches the test fixture's expected body.**
  - **Where:** `apps/web/tests/e2e/mobile-action-item-section-move.spec.ts:98-119`. The "swipe gesture POSTs to /moves" test uses `page.evaluate(async () => fetch(...))` to manually issue the POST from the page context, then asserts the intercepted body matches the body the test itself just constructed. This is a tautology — the test is asserting `body === body`. The real production primitive (a swipe on the action-item row that the production handler translates into the POST) is never exercised; the audit-event chain row that the server emits in response is never inspected; the from/to_section validation that the server enforces is never tested.
  - **What:** The test passes vacuously. A regression in the production swipe-handler that posts the WRONG body (or no body at all) would not fail this test. A regression in the server-side handler that emits an `action_item.moved` chain row with the WRONG from/to (or no payload at all) would not fail this test — the test only asserts a 200 response and ignores the actual chain row.
  - **Why it matters:** CLAUDE.md non-negotiable #2 + Quality Bar "Action item section moves emit audit events with from/to state." The spec was sold as proof of that invariant; it does not prove it. T-HD32 frames the mobile spec inventory as the load-bearing test layer.
  - **Fix:** Either (a) wire the test to dispatch real touch events on the row locator once the production swipeable row exposes the documented stable selector (and remove the `page.evaluate(fetch)` substitute), OR (b) tighten the assertion to fetch the resulting audit-log row via the test database hook and assert `kind === 'action_item.moved'` AND `payload.fromSection === 'new_business'` AND `payload.toSection === 'old_business'`. The current TODO comment at line 91-96 acknowledges the gap but the test as written still claims to verify the contract. At minimum, mark the test `test.fixme(...)` or `test.skip(...)` with a TODO referencing the docs/release-1-mobile-test-gaps.md row, so a clean pass does not imply a verified contract. Same posture applies to the third test in the same file (`move emits an audit event via the /moves POST`) — it only asserts `movePostFired === true`, which is just the route-handler echoing back.

- **F-S4: Capture-flow "no early upload" spec early-returns to a vacuous pass when the file input is not present.**
  - **Where:** `apps/web/tests/e2e/mobile-capture-flow.spec.ts:107-116`. If `fileInput.count() === 0`, the test asserts `outboundFileRequests.toHaveLength(0)` and returns. But at that point in the spec, the test hasn't EXERCISED anything — no file was set, no save was clicked. Asserting "no upload fired" after doing nothing is a tautology; the test passes whether or not the production capture-flow respects the no-early-upload invariant.
  - **What:** The spec's headline invariant ("no image bytes upload before Save") becomes unverified whenever the test fixture doesn't expose a stable file-input selector. CI-green on a regression. The comment at line 110-114 documents the gap honestly but the test still claims to assert the invariant.
  - **Why it matters:** This is the spec referenced by T-HD30 / T-HD31 as the "Capture-to-Record" signature interaction's network-level audit. Non-negotiable #4 (privacy-by-default, no early data flow). Also, the request matcher is narrow — only `/api/evidence` or `/api/uploads` (line 92-94). If a future regression posted to `/api/hazards` with the photo bytes inlined, the matcher would miss it entirely.
  - **Fix:** (a) Change the early-return path to `test.fixme(...)` so a missing fixture explicitly does not pass. (b) Broaden the request-tracker assertion: instead of matching specific endpoints, track ALL POST/PUT requests whose body is larger than N bytes (e.g., 512 bytes — larger than any auth/session beacon, smaller than even the tiny 67-byte PNG fixture; pick a threshold like 5 KB that bounds real image data). Assert no such body fires before Save — this catches `/api/hazards`-renamed regressions and any future endpoint that an attacker route could shadow.

- **F-S5: Prototype-pollution snapshot check is length-only, not content-equal — a delete-then-add attack of equal cardinality would pass.**
  - **Where:** `packages/excel-import/test/fuzz/fuzz.test.ts:295-301` (per-case check) and lines 105-108 (afterAll check uses `toEqual` on the sorted name array, which IS content-aware — so the suite-level check is correct, but the per-case check is length-only).
  - **What:** The per-case prototype-pollution scenario assertion compares `live.length !== objectProtoSnapshot.length`. A clever attack that deletes one own-name and adds another would pass this check. The afterAll check uses `toEqual` on sorted names, so it WOULD catch the asymmetric mutation — but only after all 1000 cases have run, so the failure won't identify WHICH case triggered the mutation. Combined with the `failures/` directory being non-gitignored (F-S1), reproducing a per-case prototype attack from the suite-level failure would require re-running with binary-search.
  - **Why it matters:** Non-negotiable #11 (Excel imports sanitized) + T-HD18 (silent corruption). The Fuzzing-findings document F-3 acknowledges the value-mutation gap honestly but does NOT acknowledge the length-only gap — the doc says snapshot is "own-property-name lists" and asserts equality, which is true at the suite level but not at the per-case level.
  - **Fix:** Change the per-case check on line 296 to `JSON.stringify(snapshotPrototype(...)) !== JSON.stringify(objectProtoSnapshot)` (or `live.join(',')`-based equality) so a length-equal mutation is caught per-case AND attributable to its case id. Also extend the per-case check to also snapshot `Array.prototype` and `String.prototype` (the suite-level afterAll covers these, but per-case attribution is lost). Then update `release-1-fuzzing-findings.md` F-3 to reflect the new posture.

- **F-S6: Mobile-helpers `installAuthMocks` stubs `stepUp.active: false` AND `until: null` — this is a session shape that production NEVER serves; it makes step-up freshness regressions structurally untestable from the mobile specs.**
  - **Where:** `apps/web/tests/e2e/mobile-helpers.ts:26-37`. Every mobile spec that imports `installAuthMocks` receives a session whose step-up state is permanently absent. The mocked session also returns `userId: 'mobile-e2e-user'` — a fixed identity, regardless of which test is running.
  - **What:** The mobile specs cannot exercise step-up freshness regressions. A regression that, for example, set `stepUp.active: true` indefinitely (effectively disabling step-up) would not surface in any mobile spec because the mock always overrides. Same goes for the action-bound step-up cosmetic-only divergence — the mock prevents the test from ever seeing a step-up modal, so any future fix that flips the cosmetic binding to load-bearing would not regress in the mobile specs.
  - **Why it matters:** T-HD3.5/T-HD3.6 in the security checklist (step-up freshness) — the checklist's items 3.5 and 3.8 are documented as manual-walk items, which is fine, but the mobile Playwright spec inventory was sold as covering the load-bearing flows. Step-up gating IS load-bearing per non-negotiable #16 (exports) and §3.4 of the checklist.
  - **Fix:** Add at least one mobile spec that does NOT install `installAuthMocks` and instead drives the step-up modal end-to-end (TOTP fallback path is scriptable; the spec exists for `mobile-capture-flow` flow's permission grants — the same shape applies). OR document the gap explicitly in `docs/release-1-mobile-test-gaps.md` with a "lands in" pointer and add a row to the security pre-launch checklist that the mobile-spec step-up posture is documented divergence. Currently neither is done.

### MEDIUM (document or defer)

- **F-S7: `--full` JSON report schema is not formally documented anywhere — operators consuming it have only ADR-0011 §3.7's example object and the test file's `JSON.parse` round-trip.** The runbook references `payloadShapeMismatches: []` and `gaps: []` (release-1-backup-restore-runbook §2.3, release-1-deploy-runbook §4.7) but does NOT enumerate every field the consumer can rely on (e.g., `counts.actorTimestampRegressions` is not mentioned in either runbook even though it's the dominant gap-2 signal). **Fix:** Add a "Report schema" section to `docs/release-1-audit-verify-gaps.md` (or a new `docs/release-1-audit-verify-report-schema.md`) that enumerates the JSON shape with TypeScript-style field annotations, and add a CI test that compares the test's `renderJsonReport` output against a committed golden fixture so the schema cannot drift silently.

- **F-S8: The hash-link tamper test in `audit-log-verify.test.ts:212-312` correctly exercises the "attacker mutates payload + recomputes this_hash, breaks downstream" case, but the comment on line 269-270 admits the test ASSERTS `report.ok === true` for the case where the chain is internally consistent under the attacker's recompute (truncated at row 2). This is documented behavior but it is also the highest-risk class of attack on a hash chain: an attacker with DB write access who recomputes the entire downstream chain produces a structurally valid chain that `--full` accepts. The mitigation is the external trust anchor (chain root in signed PDFs per non-negotiable #16), but that anchor's verification is NOT part of `--full`. **Fix:\*\* Add a row to the pre-launch checklist (§2) that the operator must, before deploy, capture the current chain's `highestIdx` + `this_hash` and store it in the cold-storage envelope; on every restore drill, the operator must check that the restored chain's root matches the cold-storage anchor. Currently the runbook §2.3 just says "chain on branch should match" — it does not anchor the chain root anywhere outside the chain itself.
