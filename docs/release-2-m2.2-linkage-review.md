# M2.2 Linkage Review

Reviewer: independent S5 (linkage)
Reviewed commits: 1f925e7 (S0), 42012d2 (S1), 11d8e62 (S2), b7455f9 + 5adcea5 (S3)
Scope: §A closure FK + CHECK, §B selfAttestation, §C snapshot dedupe, §D cross-meeting state, §E cross-chain anchor, §F audit-verify Gate 5, §G reopen, §H action_item_moves, §I metrics single source, §J signing_key consistency

## Findings

### CRITICAL

- **F-L1: Reopen + re-close is structurally blocked by `action_item_closures_action_item_unique`.**
  - Where: `migrations/0012_in_meeting_action_items.sql:161-162` (UNIQUE on `action_item_id`); `apps/api/src/routes/action-items/close-verification.ts:350-357,294-302,525-531`.
  - What: ADR §3.1 lines 61-62 + §3.2 line 83 + S1 brief say a re-opened item can be re-closed (creating a NEW closure row with NEW id, previous row preserved). The migration creates `CREATE UNIQUE INDEX action_item_closures_action_item_unique ON action_item_closures(action_item_id)` — a hard, full UNIQUE (not "WHERE not superseded"). The reopen route clears `action_items.closure_verification_id` but leaves the prior closure row, then a subsequent `POST /close-verification` either trips the pre-INSERT FK gate at line 294 (returns 409 ALREADY_CLOSED — incorrect because the FK is NULL) or falls through and trips the UNIQUE at line 351 (also 409 ALREADY_CLOSED). The route's own comment at lines 525-531 acknowledges "Re-closing requires a fresh route call (no second closure row is allowed by the UNIQUE)" — i.e. the design as implemented forbids re-closing entirely.
  - Why (non-negotiable #12 + T-IM4): A reopened item that the rep eventually re-verifies has NO path through the verified-close flow. The DB CHECK on `action_items` forbids `status='Closed'` without a `closure_verification_id`, AND the only route that sets this column is `/close-verification`, AND that route cannot create a second row for the same `action_item_id`. The item is permanently un-closable. SECURITY.md T-IM32 mitigation also references a `superseded_at` + partial UNIQUE pattern that was NOT implemented.
  - Fix: Either (a) relax the UNIQUE to partial `WHERE id = (SELECT closure_verification_id FROM action_items WHERE action_items.id = action_item_closures.action_item_id)` — hard in Postgres — or more practically (b) drop the full UNIQUE and add `superseded_at timestamptz` + `superseded_by_reopen_event_idx bigint` columns + partial UNIQUE `WHERE superseded_at IS NULL` (the T-IM32 SECURITY.md mitigation as originally written). The reopen route then sets `superseded_at = now()` on the prior closure row inside its transaction.

### HIGH

- **F-L2: Live metrics single-source-of-truth is violated — adjourn route hand-rolls a different `itemsClosed` formula than `computeMeetingLiveMetrics`.**
  - Where: `apps/api/src/routes/meetings/index.ts:1609-1746` (adjourn metrics dict) vs `apps/api/src/lib/compute-meeting-live-metrics.ts:143-201` + `apps/api/src/routes/meetings/metrics.ts:150`.
  - What: ADR §3.4 + the helper's docstring ("Single source of truth shared by the live metrics endpoint AND the adjournment route's metrics emit … no drift between the runtime dashboard and the canonical evidentiary record") promise both paths call the same helper. The adjourn route does NOT import or call `computeMeetingLiveMetrics`. Instead it (a) computes `itemsClosed` as `COUNT(action_items WHERE meeting_id = $1 AND status IN ('Closed','Cancelled'))` — counts Cancelled items as "closed", uses the mutable `meeting_id` operational column rather than `action_item_closures.meeting_id`; the live-metrics helper computes `itemsClosed` as the count of `action_item_closures` rows scoped to the meeting (verified closures only, no Cancelled). (b) emits `quorumCompliance.metAtCallToOrder` only; the helper emits BOTH `metAtCallToOrder` + `currentlyMet`. (c) the `meeting.adjourned` payload omits the helper's `closureVerifications: {total,selfAttestation,peerVerified}` shape — adjourn emits a different `closureVerifications` shape that IS aligned, but it is computed independently rather than via the helper.
  - Why (non-negotiable #2): The dashboard chip displayed to the rep mid-meeting will quote a different `itemsClosed` than the chain-anchored adjournment record. An arbitrator comparing the rep's screenshot to the chain-anchored payload sees drift.
  - Fix: Refactor the adjourn route to fetch the same row shapes the metrics endpoint fetches and call `computeMeetingLiveMetrics(...)` with `nowMs = actualEndAt.getTime()`, then use the result for the `meeting.adjourned` payload. The metrics endpoint and adjourn now share a single formula by construction.

- **F-L3: `meeting-history` endpoint promised in ADR §3.2 + §3.7 was never shipped; S3 web client builds a degraded client-side timeline.**
  - Where: ADR §3.2 specifies `GET /api/action-items/:id/meeting-history` returning per-meeting `moves[]`, `snapshots[]`, and `closure?` — the canonical cross-meeting state view. No such route exists (grep confirms zero hits in `apps/api`). `apps/web/src/views/action-item-detail-view.tsx:600-706` `MeetingHistoryTimeline` instead derives entries client-side from the existing move-history payload, surfacing only `meetingId`, `firstSeenAt`, `lastTouchedAt`, `moveCount`.
  - Why (non-negotiable #12, T-IM34/35, ADR §3.7): The "raised in meeting A, worked in B, closed in C" cross-meeting state visibility is not realised. The detail view does NOT show the status/section the item had at each meeting's close (sourced from `meeting_action_item_state` finalized snapshots), does NOT show the closure attestation in the timeline, and CANNOT show meetings where the item was touched only via status PATCH (no move row) because the client-side derivation hangs off moves only. An item raised in A, status-changed twice in B without a section move, then closed in C, shows only A and C in the timeline.
  - Fix: Implement `GET /api/action-items/:id/meeting-history` per ADR §3.2 response shape, joining `action_item_moves` ∪ `meeting_action_item_state` ∪ `action_item_closures` by meeting_id. Update the `MeetingHistoryTimeline` to consume the new payload.

- **F-L4: `writeLiveActionItemSnapshot` does not handle the TM-fold-2 partial-UNIQUE — duplicate (status,section) PATCH retry will 500 the whole transaction.**
  - Where: `apps/api/src/routes/meetings/index.ts:2238-2251`; the partial UNIQUE `meeting_action_item_state_live_dedupe_unique` from `migrations/0012_in_meeting_action_items.sql:223-226`; called from `apps/api/src/routes/action-items/index.ts:1084` (PATCH), `:387` (create), and `apps/api/src/routes/action-items/close-verification.ts:401` (close-verification).
  - What: The helper INSERTs a fresh snapshot row with no `ON CONFLICT` clause and no try/catch. The partial UNIQUE will raise on the second idempotent PATCH that lands the same `(meeting_id, action_item_id, snapshot_status, snapshot_section)` combination while `snapshot_kind='live'`. The thrown error bubbles up through `db.transaction(...)`, rolling back the entire PATCH or close-verification — including the `action_items` UPDATE, the `action_item.updated`/`action_item.status_changed` chain rows, and (in the closure path) the `action_item_closures` INSERT and the closure chain anchor. By contrast, the adjourn finalize path at `apps/api/src/routes/meetings/index.ts:1671-1713` correctly wraps the parallel INSERT in try/catch and skips on UNIQUE collision.
  - Why (T-IM7, T-IM11, TM-fold-2): The fold was explicitly designed to make idempotent retries silent ("silent dedupe is the desired behavior — same logical state = no new row"). As-shipped it converts retries into rollbacks. A flaky mobile network during in-meeting status updates will surface as random PATCH 500s after the second attempt at the same logical state.
  - Fix: Wrap the INSERT in try/catch (or use `INSERT … ON CONFLICT … DO NOTHING RETURNING id` and short-circuit when no row returned). Match the adjourn-path pattern at lines 1703-1712. When the INSERT is a dedupe no-op, also skip the `append(...)` chain emission so the chain doesn't carry a snapshot row pointing at no DB row.

### MEDIUM

- **F-L5: Closure with `meetingId` pointing at a non-`in_progress` meeting silently binds; only the cross-anchor + snapshot are skipped.**
  - Where: `apps/api/src/routes/action-items/close-verification.ts:307-325` (only checks status to decide cross-anchor) and `:338` (INSERT always uses body.meetingId).
  - What: The close-verification route binds `action_item_closures.meeting_id = body.meetingId` unconditionally. Only the cross-anchor + live-snapshot at lines 396-440 are gated on `meeting.status='in_progress'`. A POST with `meetingId=<finalized_meeting_id>` writes a closure row pretending to belong to that meeting, but emits no `meeting.action_item_status_changed` cross-anchor. Gate 5 verified `meeting.finalized`'s `closureVerificationCount` payload AT finalize-time, but the DB COUNT now drifts. The PATCH route at `routes/action-items/index.ts:748-768` correctly rejects `meetingId` pointing at frozen meetings with 422 `MEETING_FROZEN`; the close-verification route does not have the parallel gate.
  - Why (T-IM36 parity): A closure that claims meeting-context but emits no cross-anchor cannot be audit-verified end-to-end. The Gate 5 invariant `payload.closureVerificationCount == count_of_chain_events` stays true only because closures-after-finalize don't have a chain event in the window — but the DB row count for `action_item_closures WHERE meeting_id = $X` will exceed `closureVerificationCount` permanently.
  - Fix: Mirror the PATCH route's frozen-meeting gate. If `body.meetingId` is set, SELECT the meeting status and reject with 422 `MEETING_FROZEN` (or coerce to NULL and proceed without meeting binding) when the meeting is not `in_progress`.

- **F-L6: `action_item_closures.action_item_id` FK is `ON DELETE CASCADE` — closure rows can vanish silently.**
  - Where: `migrations/0012_in_meeting_action_items.sql:141-143`.
  - What: ADR §3.1 line 38 specifies `ON DELETE RESTRICT ON UPDATE RESTRICT` (append-only invariant on the attestation row). The migration ships `ON UPDATE RESTRICT ON DELETE CASCADE`. If an action_items row is ever deleted (admin script, future GC), the closure attestation evaporates, but its `action_item.closure_verified` chain row remains. Gate 5's verifier sees the chain event but the corresponding DB row is gone. Worse, the `meeting.finalized` payload's `closureVerificationCount` would drift from `SELECT COUNT(*) FROM action_item_closures WHERE meeting_id = $1`.
  - Why (non-negotiable #2, T-IM31): The closure attestation is the evidentiary anchor — the chain payload carries the closure's canonical-row hash + the workplace-key signature. Deleting the row destroys the ability to re-verify the signature.
  - Fix: New migration that runs `ALTER TABLE action_item_closures DROP CONSTRAINT action_item_closures_action_item_fk; ALTER TABLE action_item_closures ADD CONSTRAINT action_item_closures_action_item_fk FOREIGN KEY (action_item_id) REFERENCES action_items(id) ON UPDATE RESTRICT ON DELETE RESTRICT;` per the ADR. Existing action_items deletes already trip a different FK chain so this is operationally safe.

- **F-L7: `action_item_closures_audit_idx` FK promised in ADR §3.1 was not added.**
  - Where: ADR §3.1 line 61 specifies `audit_idx bigint not null references audit_log(idx)` + `action_item_closures_audit_idx_unique`. The migration's `action_item_closures` table does NOT carry an `audit_idx` column at all, nor the corresponding unique index.
  - What: M2.1 `meeting_signatures` uses the audit_idx FK as the per-entity chain anchor pointer (ADR-0007 §3.6). Without it, given an `action_item_closures` row, there is no FK-traversal path to find the `action_item.closure_verified` chain row — the lookup must scan `audit_log` by `payload->>'closureId'`. The verifier still functions (it walks the chain, not the table) but the operational/admin discoverability + the per-entity audit-anchor invariant from ADR-0007 §3.6 is broken.
  - Why (non-negotiable #2): A closure row without a back-pointer to its chain event is harder to audit out-of-band (e.g., when a rep asks "show me the chain event for closure X").
  - Fix: Add an `audit_idx bigint NOT NULL REFERENCES audit_log(idx) ON UPDATE RESTRICT ON DELETE RESTRICT` column + a UNIQUE index on it, populate from `closureChainRow.idx` in the close-verification transaction.

### LOW

The 422 `COUNTER_SIGNER_ROLE_INVALID` is raised whenever counter-signer != auth.userId, even if the named user IS a worker_co_chair. This is correct for single-tenant scope and matches the S0 Q1 decision, but the error message is misleading once 2.5 onboards a second worker rep. - **F-L9: The cross-anchor emitted by close-verification (line 414-433) uses `kind: 'meeting.action_item_status_changed'` with `statusChangedEventHash = closure_verified.thisHash` — the verifier's Gate 4 (audit-log-verify.ts:734) correctly accepts either upstream kind, so this composes. Verified clean.** (Logged here to make explicit that Gate 4 was extended for this case.)

- **F-L10: The Section Move primitive (CLAUDE.md "Signature Interactions") for closed items has no defined behavior after F-L1.** A rep cannot reopen + re-close after F-L1 lands; consequently `completed_this_period → old_business` (the 1.6 reopen path) silently leaves the item with `status='Closed' + closure_verification_id=NULL` ... wait, that's caught by the DB CHECK and the moves route is unaffected (moves don't touch status). The `completed_this_period → old_business` move would change section only; status stays Closed; the bi-directional CHECK still passes. No defect from M2.2's perspective. Verified clean.

## Verified clean

- §A closure FK + CHECK: The CHECK `(status='Closed' AND closure_verification_id IS NOT NULL) OR (status != 'Closed' AND closure_verification_id IS NULL)` is exact (migrations/0012:198-203). The close-verification route INSERTs closure → UPDATEs action_items in a single `db.transaction` (close-verification.ts:273-441). Direct PATCH to `status='Closed'` is rejected at routes/action-items/index.ts:713-722 with 422 CLOSE_VIA_VERIFICATION. Direct PATCH to ANY status when current is Closed is rejected at routes/action-items/index.ts:725-740 with 422 REOPEN_VIA_REOPEN. The DB CHECK is the structural backstop against direct SQL.
- §B selfAttestation: Both DB CHECKs present (migrations/0012:131-140). Route layer enforces the same invariant at close-verification.ts:164-183 with clean 422 INVALID_SELF_ATTESTATION_FLAG BEFORE the INSERT — defense in depth landed correctly.
- §E cross-chain anchor: Close-verification emits `action_item.closure_verified` THEN `meeting.action_item_status_changed` in the same `db.transaction` (close-verification.ts:373-440). The cross-anchor's `statusChangedEventHash` is the closure event's `thisHash` (line 413), not the action_item.status_changed kind's hash — Gate 4 (audit-log-verify.ts:734) accepts either via `statusChangedByHash.get(claimedHash) ?? closureByHash.get(claimedHash)`. Same posture in the PATCH route at routes/action-items/index.ts:996-1054 where status_changed → cross-anchor share the per-item event hash.
- §F audit-verify Gate 5 (excluding F-L5/F-L6 hazards): Correctly filters closures by `payload.meetingId` between `meeting.created.idx` and `meeting.finalized.idx`. Closures with `payload.meetingId === null` (closure outside meeting) are correctly excluded at audit-log-verify.ts:390-394. The reopened-then-reclosed edge is moot under F-L1.
- §H action_item_moves: The `meeting_id` column was added in 1.6 (schema.ts:513); the meeting FK was promoted to actual FK in migrations/0011_meeting_lifecycle.sql:552-555. M2.2 added nothing; the per-meeting move read is `WHERE meeting_id = $1` per S0 Q5 (verified in the meeting-detail-view consumption).
- §J signing_key consistency: Close-verification.ts:307-325 enforces SIGNING_KEY_REBOUND 422 mirroring M2.1 F-L2 — first-prior-closure's signing_key_id must match the active signing key. Inside-meeting closures cannot rebind. Verified clean.

## Verdict

Four serious defects (one CRITICAL re-closure dead-end; HIGH metric drift between live + adjourned; HIGH missing cross-meeting endpoint; HIGH snapshot dedupe converts retries into rollbacks) plus three MEDIUM closure-row integrity gaps require fixes before merge to preserve non-negotiables #2 and #12; the closure-verification chain-of-custody otherwise hangs together end-to-end with appropriate transactional + signing-key + step-up discipline.
