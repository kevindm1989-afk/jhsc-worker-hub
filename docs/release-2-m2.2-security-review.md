### HIGH (fix in S5 bundle)

- F-S1: `writeLiveActionItemSnapshot` does not dedupe — TM-fold-2 partial UNIQUE raises an uncaught exception that rolls back the entire status PATCH transaction.
  - **Where:** `apps/api/src/routes/meetings/index.ts:2218-2268` (helper); `apps/api/src/routes/action-items/index.ts:1083-1093` (caller in PATCH); `apps/api/src/routes/action-items/close-verification.ts:401-409` (caller in close-verification).
  - **What:** The migration adds `meeting_action_item_state_live_dedupe_unique` partial UNIQUE on (`meeting_id`, `action_item_id`, `snapshot_status`, `snapshot_section`) WHERE `snapshot_kind='live'` (migration 0012:223-226). The writer issues a bare `INSERT … VALUES …` with no `ON CONFLICT` clause, no try/catch, and no MAX-snapshot pre-read dedupe. When two PATCHes from the same rep — or a 1.10 retry that defeated idempotency middleware — land the same (status, section) for the same (meeting, action_item), the second INSERT raises a UNIQUE violation that bubbles up, the transaction rolls back, and the user-visible `PATCH /api/action-items/:id` returns 500. The intended same-state dedupe path described in SECURITY §2.14 T-IM11 ("the writer dedupes against the most-recent live snapshot — if `(snapshot_status, snapshot_section, assignee_hash)` matches the row at `MAX(snapshot_at)` … the writer no-ops and emits no chain row") was never implemented in S1/S2.
  - **Why it matters:** T-IM7 + T-IM11 (status-change retry / idempotency-key reuse cascades) — the partial UNIQUE is doing its job but the route surface treats it as a 500. A flaky network or a multi-tab rep can churn through this case routinely. Worse, because the entire `db.transaction` rolls back, the legitimate status change + per-item `action_item.status_changed` + cross-anchor `meeting.action_item_status_changed` are ALL discarded — the rep sees a 500, the chain has no record, and the action item's status stays at the pre-PATCH value. The route silently regresses to "lossy under retry" under exactly the offline-meeting-room scenario the mobile-primary design targets.
  - **Fix:** Add `ON CONFLICT (meeting_id, action_item_id, snapshot_status, snapshot_section) WHERE snapshot_kind='live' DO NOTHING` to the INSERT and short-circuit the chain anchor emit when `RETURNING id` returns empty; OR wrap the INSERT in a try/catch on `/meeting_action_item_state_live_dedupe_unique/.test(e.message)` mirroring the adjourn route's pattern at `apps/api/src/routes/meetings/index.ts:1703-1713`. The chain anchor should not fire on a deduped no-op insert.

- F-S2: `action_item.closure_verified` chain payload omits `closureReasonHash` and `closedAt`, defeating offline chain-only signature verification.
  - **Where:** `apps/api/src/routes/action-items/close-verification.ts:373-389` (chain emit); compare canonical at `apps/api/src/lib/compute-action-item-closure-canonical.ts:44-58` and SECURITY §2.14 T-IM5 mitigation text.
  - **What:** The Ed25519 attestation canonical includes `closureReasonHash`, `closedAt`, `counterSignedAt`. The chain payload emitted at lines 375-386 includes only `actionItemId, closureId, meetingId, closerActorId, counterSignerActorId, selfAttestation, signingKeyId, evidenceHash, attestationSigHash`. SECURITY §2.14 T-IM5 explicitly says the chain payload "carries `{actionItemId, verifierActorId, verificationId, narrativeHash, attestationSigHash, verifiedAt}`" — both `narrativeHash` (= `closureReasonHash`) and `verifiedAt` (= `closedAt`) are missing.
  - **Why it matters:** T-IM5 + T-IM33 — an offline verifier walking only the chain cannot independently re-derive the canonical digest the Ed25519 signature attests over (the signature is opaque without `closedAt` + `closureReasonHash`). They have to fetch the `action_item_closures` row to reconstruct what was signed, which forfeits the chain's standalone tamper-detection property. Worse, if a hostile DB-level attacker mutates `closure_reason_envelope_ct` after the row is signed, the chain payload has no `closureReasonHash` snapshot to detect the swap; the only detection path is "re-sign and compare" (requires the private key).
  - **Fix:** Extend the chain payload at `close-verification.ts:375-386` and the discriminated-union type at `packages/shared-types/src/index.ts:960-975` with `closureReasonHash: string` and `closedAt: string` (+ `counterSignedAt: string` for completeness). Mirror in audit-payload type tests.

- F-S3: Live snapshot section is hardcoded incorrectly in the closure route — finalized adjournment snapshot will record `new_business` for items that were actually in `old_business` or `recommendation`.
  - **Where:** `apps/api/src/routes/action-items/close-verification.ts:401-409` (snapshot call) and the FOR UPDATE projection at `:277-286`.
  - **What:** The FOR UPDATE SELECT at line 277 reads `id, status, closure_verification_id, meeting_id, version` — `section` is NOT projected. The snapshot write at line 406 passes `section: item.status === 'Closed' ? 'completed_this_period' : 'new_business'`. Because the pre-update status check at line 294 rejects any item already `Closed`, the ternary's true branch is unreachable; the section is unconditionally `'new_business'`. An action item closed from `old_business` lands a `live` snapshot saying `new_business`; at adjournment the `DISTINCT ON (action_item_id) ORDER BY snapshot_at DESC` promotion picks this most-recent live row and stamps the `finalized` snapshot with the wrong section.
  - **Why it matters:** T-IM30 / T-ML16 — the `meeting.action_item_snapshot` chain payload + the `finalized` snapshot row that backs the M2.3 PDF generator carries demonstrably-incorrect provenance. An arbitrator reading "action item #42 was closed from `new_business`" when the move history shows it was carried in `old_business` for three quarters surfaces a chain-vs-history contradiction that hurts evidentiary weight. The verifier's `--check-action-items` Gate 4 only checks the hash linkage, not the section value, so this drifts silently.
  - **Fix:** Add `section` to the FOR UPDATE projection at `:277-286` and pass `section: item.section as ActionItemSection` to `writeLiveActionItemSnapshot`. Drop the false-positive ternary.

### MEDIUM

- F-S4: Reopen + re-close path is structurally blocked by `action_item_closures_action_item_unique`, contradicting the ADR's "append-only history with new row per re-close" stance.
  - **Where:** `migrations/0012_in_meeting_action_items.sql:161-162` (UNIQUE on `action_item_id`); `apps/api/src/routes/action-items/close-verification.ts:519-531` (acknowledgment comment).
  - **What:** The UNIQUE index makes at most ONE closure row per action item ever. The route's own comment block at lines 525-531 admits this: "Wait — UNIQUE on (action_item_id) means there can be exactly ONE closure row per item ever … Re-closing requires a fresh route call (no second closure row is allowed by the UNIQUE; the re-open chain anchor + the existing closure row are the operational trace)." A re-close attempt after reopen will trip the UNIQUE in the route's INSERT and surface as `ALREADY_CLOSED` 409 — but the original closure row's content (closer, counter-signer, closed_at, reason envelope) is FROZEN from the first close, even though the item has been re-evaluated. The reopen / re-close cycle produces an `action_item.reopened` chain row pointing at the old `previousClosureId` but no `action_item.closure_verified` for the second close. This deviates from ADR-0013 §3.5 ("Re-closing requires a new route call … A subsequent re-close MUST go through a new route … `POST /api/action-items/:id/close-verification` is idempotent on the UNIQUE — see §3.2") and from the prompt's expectation ("re-closing requires a new row + the FK swap; the OLD closure stays referenced from the chain"). Either the migration is wrong (UNIQUE should be partial WHERE `superseded_at IS NULL` as SECURITY §2.14 T-IM32 mitigation actually specifies) or the route's reopen path should fail-fast on items that already have a closure row.
  - **Why it matters:** T-IM4 / T-IM32 — the chain captures `action_item.reopened` but cannot anchor the second close. An arbitrator reading the chain sees `closure_verified (idx N) → reopened (idx N+50) → ???` with no second closure anchor, no second counter-sign timestamp, no second basis-of-closure ciphertext. The single closure row's `closed_at` predates the reopen event, and the row's `verified_by_jhsc_id` was nulled out on reopen (line 536) — so the `action_items.verified_by_jhsc_id` is NULL but the row's status is `Closed` after the (impossible) re-close. The CHECK constraint `(status='Closed') == (closure_verification_id IS NOT NULL)` can't even be satisfied for re-close because the FK can't be re-pointed to a new row that doesn't exist.
  - **Fix:** Two paths. (a) Match T-IM32 by making the UNIQUE partial: `WHERE superseded_at IS NULL` (requires adding `superseded_at TIMESTAMPTZ NULL` to the schema and updating reopen to stamp it). (b) Or, accept the single-closure-per-item invariant explicitly: change `action_item_closures_action_item_unique` to a comment-only documentation row, document the limitation in the ADR + runbook, and have reopen surface a "re-closing requires admin intervention" 422 with a clear message. Choose (a) for fidelity to T-IM32's mitigation text.

- F-S5: `meeting.adjourned` payload type definition (`packages/shared-types/src/index.ts:880-895`) does not include `closureVerifications`, but the route at `apps/api/src/routes/meetings/index.ts:1741-1745` emits it inside the metrics dict.
  - **Where:** `packages/shared-types/src/index.ts:880-895` (discriminated-union shape); `apps/api/src/routes/meetings/index.ts:1731-1758` (emission).
  - **What:** The discriminated-union shape for `meeting.adjourned.metrics` enumerates `durationSeconds, itemsRaised, itemsClosed, recommendationsDrafted, inspectionsReviewed, quorumCompliance` but not `closureVerifications`. The route appends `closureVerifications: { total, selfAttestation, peerVerified }` to the dict regardless. The `append()` helper is loose enough that runtime succeeds, but the audit-verify `--check-meetings` Gate 2 structural check at `apps/api/scripts/audit-log-verify.ts:514-518` only validates the required fields, so a malformed extension would silently pass. The S0 addendum specifies the additive shape on `meeting.finalized` (TM-fold-5) — which IS typed at `:910-921` — but the parallel addition to `meeting.adjourned` was skipped.
  - **Why it matters:** T-AC9 / T-IM27 (chain-payload PI discipline + drift) — the shared-types discriminated union is the project's structural defense against payload drift. If the type doesn't enumerate `closureVerifications`, a future contributor could add `selfAttestationActorIds: string[]` to the dict without a typecheck failure, and the M2.2 promise of "AGGREGATE-ONLY, no per-actor breakdown" (T-IM27) erodes silently. Today the field carries only counts (verified) but the type-level gate is the structural backstop.
  - **Fix:** Extend the `meeting.adjourned.metrics` shape at `packages/shared-types/src/index.ts:884-894` with `readonly closureVerifications: { readonly total: number; readonly selfAttestation: number; readonly peerVerified: number }` (mark optional with `?` if pre-2.2 finalized rows from M2.1 lack the field; otherwise required). Refresh the audit-payload type-guard tests.

- F-S6: `previousClosureId` re-use across reopen + re-close cycles creates a chain that points "forward" to a stale closure row — verifier Gate 2 cannot detect it.
  - **Where:** `apps/api/src/routes/action-items/close-verification.ts:516-554` (reopen); audit-verify Gate 2 at `apps/api/scripts/audit-log-verify.ts:689-701`.
  - **What:** Connected to F-S4. On reopen, the chain emits `action_item.reopened {previousClosureId, …}` pointing at the original closure row. If a re-close DID succeed (today it can't, see F-S4, but if F-S4 is fixed with partial UNIQUE), the chain would have:
    1. `closure_verified` (idx A) — pointer to closure row C1
    2. `reopened` (idx B) — `previousClosureId: C1`
    3. `closure_verified` (idx C) — pointer to closure row C2
    4. `reopened` (idx D) — `previousClosureId: ???`
  - Gate 2 (`reopened_no_prior_closure`) only checks the FIRST closure-verified upstream by `actionItemId`. A reopen at step 4 currently passes Gate 2 by matching the idx-A closure (since `closureById.set` at line 664 overwrites — actually `set` on duplicate IDs keeps the latest, but only ONE entry per actionItemId, so it stores C2). The verifier loses the distinction between "the first close was reopened" vs "the second close was reopened"; a chain analyst cannot reconstruct the sequence.
  - **Why it matters:** T-IM4 — the chain semantics fail to compose under repeated lifecycle events. Even with the F-S4 fix, the verifier needs to gate "reopened.previousClosureId points at the MOST-RECENT closure_verified upstream for this actionItemId, not just any upstream."
  - **Fix:** Gate 2 should resolve `previousClosureId` to the chain row whose `payload.closureId` matches, and assert that closure row's idx > all OTHER closure_verified events for this actionItemId that precede the reopen. Today the gate doesn't read `previousClosureId` at all — it only checks that SOME closure_verified is upstream by actionItemId. Tighten the check.

### LOW

- F-S7: Step-up freshness is structurally only an (actor, time) tuple — `action='action_item.close_verification'` is decorative on the server side per the documented honest contract.
  - **Where:** `apps/api/src/auth/step-up.ts:73-101` (honest contract block); `apps/api/src/routes/action-items/close-verification.ts:120-129`.
  - **What:** `checkStepUpFreshness` does NOT actually verify the `action` claim matches the called endpoint. Per the documented honest contract at `step-up.ts:73-87`: "A fresh step-up grant obtained for ANY action (e.g. `recommendation.read`, `inspection.finding.read`, `passkey.remove`) is accepted by this function for ANY OTHER action within the freshness window." So T-IM2's mitigation ("a grant for `meeting.sign.worker_co_chair` returns 401-StepUp at this endpoint even if within the 60s window" + "Integration test asserts a grant for `meeting.sign.worker_co_chair` returns 401-StepUp") is structurally false on the implementation. Also T-IM41's deepening fold ("the WebAuthn `clientDataJSON` challenge … includes `actionItemId` AND the action item's `version`") is unimplemented.
  - **Why it matters:** T-IM2 / T-IM41 — the documented mitigations in SECURITY §2.14 promise per-action and per-resource binding that the substrate doesn't enforce. A rep who step-ups for a recommendation read could replay the grant within 60s against a closure verification on any action item. The 60s freshness window is real (and short), and the cryptographic attestation captures the closer/counter-signer identities — so the residual is narrow — but the SECURITY narrative overstates the protection.
  - **Fix:** This is a known forward-seam (documented in `step-up.ts:82-87` as a "1.12 hardening item"). Either close it in the S5 bundle by extending the step-up grant to carry `(action, until, resource_id?)` tuples and tightening the freshness check, or update SECURITY §2.14 T-IM2 / T-IM41 to honestly mark the per-action and per-resource binding as forward-seam rather than mitigation. Recommend the latter for M2.2 scope; track the per-action binding as the dedicated hardening milestone.

- F-S8: Reopen route emits `action_item.reopened` but does not write any cross-chain `meeting.*` anchor — a reopen during a live meeting has no meeting-context chain envelope.
  - **Where:** `apps/api/src/routes/action-items/close-verification.ts:542-554`.
  - **What:** The PATCH path (`action-items/index.ts:1019-1054`) emits `meeting.action_item_status_changed` cross-anchors on every in-meeting status change; the close-verification path does the same (`close-verification.ts:413-433`). The reopen path does NEITHER — even when an item is reopened during an `in_progress` meeting (a plausible workflow: a closure verified earlier in the meeting is reconsidered later in the same meeting), no `meeting.action_item_status_changed` cross-anchor is emitted.
  - **Why it matters:** T-IM4 / T-IM13 — the verifier `--check-meetings` Gate 5 cross-checks `meeting.finalized.closureVerificationCount` against the count of `closure_verified` events, but doesn't expect a parallel "reopens-during-meeting" count. An arbitrator walking the meeting's chain envelope sees the closure but not the in-meeting reopen, even though the closure was effectively undone within the same meeting. The provenance gap.
  - **Fix:** Extend the reopen route to emit `meeting.action_item_status_changed {meetingId, actionItemId, fromStatus: 'Closed', toStatus: 'In Progress', changedAt, statusChangedEventHash: <reopened-row-hash>}` when the item's `meeting_id` points at an `in_progress` meeting at reopen time. Mirror the helper pattern.

- F-S9: Counter-signer role check is a string equality (`counterSignerActorId === auth.userId`), not a role-table lookup — the documented "worker_co_chair role" gate is collapsed to "is the authenticated rep."
  - **Where:** `apps/api/src/routes/action-items/close-verification.ts:185-200`.
  - **What:** The Q1 user decision (ADR §S0 addendum) is "M2.2 ships co-chair-only counter-sign" with a 2.5 forward-seam for a second worker rep. The route's role gate is documented at lines 185-190 as collapsing to `counterSignerActorId === auth.userId` "until 2.5 introduces a second in-app worker rep." There is NO query against a user_roles table; no resolution that the auth.userId actually IS the worker_co_chair. Single-tenant scope makes this work today (the rep IS the sole worker_co_chair), but the comment at line 196 conflates "must equal auth.userId" with "must be a worker_co_chair." A future deploy that seeds a second worker_rep user (per 2.5) and runs without first refactoring this route will produce a `selfAttestation: false` closure where the counter-signer is whoever made the API call — not necessarily a `worker_co_chair`.
  - **Why it matters:** T-IM1 — single-tenant single-rep scope is the load-bearing assumption. The structural backstop is the DB CHECK constraint (`closed_by_actor_id != counter_signed_by_actor_id` when `self_attestation = false`). Documented residual; will require explicit refactor at 2.5 onboarding.
  - **Fix:** Add a TODO at line 185 referencing milestone 2.5: "When the second worker rep onboards, replace the equality check with a JOIN against a user_roles table filtered by `role IN minutesSignerRoles`." Surface this in the M2.2 runbook so the 2.5 implementer doesn't drift.

---

**Verdict:** No release-blockers; three HIGH findings and three MEDIUM findings should fix in the S5 bundle. The closure-verification crypto substrate (Ed25519 over canonical row + step-up + bi-directional CHECK) is structurally sound; the gaps are payload-shape omissions (F-S2, F-S5), the snapshot writer's missing dedupe (F-S1), a section-projection bug (F-S3), and the closure-row-singleton tension with the ADR's append-only stance (F-S4).
