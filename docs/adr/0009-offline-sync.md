# ADR-0009: Offline-First Sync (Dexie + sync queue + service worker + conflict resolution)

Status: Accepted, Milestone 1.10
Date: 2026-06-02
Authors: codifies Milestone 1.10 architect-phase decisions; pairs with `SECURITY.md` §2.10 (forthcoming) and `docs/runbooks/offline-sync.md` (forthcoming).

## Context

The rep does not work from a desk. The 1.5–1.9 surfaces — hazard capture, action-item moves on the minutes board, inspection findings during a freezer walk-through, recommendation drafting on the commute home — were all built behind a hard online assumption: every typed-API-client call rides a live `fetch` to the Hono API on the Fly Machine, and a dropped LTE bar surfaces as a thrown `ApiError` and a half-written form. That assumption was the deliberate 1.5–1.9 forward seam, called out in every prior ADR's "1.10 absorbs" follow-up (`docs/adr/0007-inspections.md` §"Follow-ups" — _"the capture flow's Dexie stubs (queue findings + photos for upload), conflict resolution for concurrent edits on the same inspection, signature queueing"_; `docs/adr/0008-recommendations.md` §"Follow-ups" — _"offline drafting queue (the drafting form is the obvious offline candidate — long-form text editing on a phone in a warehouse with patchy WiFi), conflict resolution if two devices edit the same draft, citation picker offline behavior"_). 1.10 is where that seam closes.

CLAUDE.md non-negotiable #9 is the structural force behind this milestone: every feature is designed for a rep on a 390px phone in a freezer aisle, on a dock that drops to 2G, or at a meeting-room table where the building WiFi sometimes resolves and sometimes doesn't. ARCHITECTURE.md §3 already declares the offline-first posture as "Read: IndexedDB first (instant), fetch fresh from API in background; Write: IndexedDB first (instant feedback), enqueue sync to API." 1.10 is the milestone where that posture stops being a stub and becomes load-bearing. The tech-stack lock in CLAUDE.md names Dexie.js (IndexedDB) as the local-first layer with a custom sync engine and `vite-plugin-pwa` as the service worker. Both choices are honored here; no stack churn.

Three things make 1.10 different from anything 1.2–1.9 has shipped:

1. **It touches every existing module.** The 1.5 hazards routes, the 1.6 action-items moves, the 1.7 evidence capture, the 1.8 inspections findings + signatures, and the 1.9 recommendations drafting all become offline-queueable in this milestone. The blast radius for the ratchet (the new `Idempotency-Key` middleware, the new `clientId` body field on every POST handler, the new `If-Match` etag on every PATCH handler) is the entire prior route surface.
2. **The chain anchor lives at SERVER drain time, not local-write time.** This is the most consequential interaction with non-negotiable #2. The chain — the tamper-evident hash chain that ADR-0002 anchors every sensitive write to — is the post-sync record. Before the sync queue drains, the local Dexie row is rep-trusted (the rep is the data custodian on their own device). After the queue drains, the chain row appears on the server with the server's `now()` timestamp. The runbook will state this in operational language: a rep typing offline at 10:14am, driving home, and seeing the app drain overnight produces a chain row at 02:47am the next morning. The chain proves "the server received this at time T," not "the rep typed it at time T'."
3. **The workplace key is server-side, which bounds what can be offline-revealed.** The encryption KEK lives in Fly Secrets (`MASTER_KEY`); the workplace public key ships to the browser via `/api/auth/session` so the browser can sealed-box-encrypt fresh writes before they hit Dexie or the queue. But every `reveal` endpoint (hazard reporter, inspection finding, evidence decrypt, recommendation body) decrypts on the server because the workplace _private_ key lives there. Offline reveal is structurally impossible, not a policy choice. The conflict-resolution flow that requires comparing decrypted plaintext is therefore online-only.

`apps/web/src/components/app-shell/app-shell.tsx` is the visual anchor (the sync-status chip lives in the top bar; the bottom tab bar carries the offline indicator). `apps/web/src/app.tsx` is the route surface this ADR plumbs Dexie under. ADR-0007 (inspections) is the closest size + cross-cutting model — same shape (data model + lifecycle + step-up + cross-cutting ratchet pattern) at a similar slice count. ROADMAP.md 1.10 scope is the Dexie.js IndexedDB layer + sync queue with retry + exponential backoff + optimistic UI + conflict resolution UI + sync status indicator + service worker with offline shell + PWA install prompt on all platforms.

## Decision

Land seven Dexie tables (`hazards`, `action_items`, `action_item_moves`, `inspections`, `inspection_templates`, `inspection_findings`, `inspection_signatures`, `recommendations`, `recommendation_citations`, `recommendation_responses`, `recommendation_action_item_links`, `evidence_files`, `legal_clauses`, `sync_queue`, `sync_conflicts`) in `apps/web/src/db/`, plus a new server-side `sync_idempotency` table for replay deduplication, plus a new `Idempotency-Key` middleware ratcheted into every existing POST/PATCH handler that creates or mutates a row, plus a new `clientId` body field accepted by every create handler so the rep's device-generated UUID becomes the canonical server id, plus a service worker generated by `vite-plugin-pwa` that caches the app shell + fonts + workplace public key + legal corpus + inspection templates and intercepts `/api/*` mutations to write to the queue when offline, plus a sync-status chip in the app shell and a conflict-resolution UI surfaced from `sync_conflicts`. All sensitive sync-queue payloads carry envelope-encrypted ciphertext written client-side under the workplace public key (same shape as the 1.7 evidence DEK encrypt path); IndexedDB is treated as untrusted-at-rest storage and no decrypted plaintext is persisted there. The chain anchor for every queued operation fires when the queue drains on the server, NOT at queue-typed-at time; the 21-day s.9(21) clock is measured from server `submitted_at`, NOT from queue-typed-at, and the UI says so. PDF exports, all `reveal` endpoints, all step-up flows, all auth flows, and first-run setup are explicitly NOT queueable — they surface "Network required" when offline. Backoff curve is exponential at 1s / 5s / 30s / 5min / 30min / 2h / 12h / 24h with a 48h dead-letter ceiling. The polymorphic-FK ratchet pattern from ADR-0007 §3.7 / ADR-0008 §3.5 extends one more time: a new `sync_op` source type is NOT added to `action_items.source_type` (the queue's payload carries the existing source_type values verbatim — `manual | hazard | recommendation | inspection`; the queue is plumbing, not provenance).

### 3.1 Dexie schema design (per-entity tables + `_sync_state` + `_local_id` + indexes)

The Dexie database is `apps/web/src/db/dexie.ts` exporting a single `db` singleton. Schema version 1; bumps follow the Dexie idiom (`db.version(N).stores({...}).upgrade(tx => ...)`) — append-only forward migrations only, same posture as CLAUDE.md "migrations are append-only" for the server side.

Fifteen tables in version 1: `hazards`, `action_items`, `action_item_moves`, `inspections`, `inspection_templates` (read-only cache), `inspection_findings`, `inspection_signatures`, `recommendations`, `recommendation_citations`, `recommendation_responses`, `recommendation_action_item_links`, `evidence_files`, `legal_clauses` (read-only cache), `sync_queue`, `sync_conflicts`. Per-table indexes are chosen for the views' actual read patterns: `hazards` indexes `_sync_state, status, severity, created_at`; `action_items` indexes `_sync_state, section, status, meeting_id, source_type`; `inspection_findings` indexes `_sync_state, inspection_id, section_number, status`; `sync_queue` indexes `next_attempt_at` (queue worker hot path — `.where('next_attempt_at').belowOrEqual(now()).first()`); every entity table indexes `_sync_state` because the sync-status chip counts non-`clean` rows across all tables in one read (§3.11).

**Every mutable entity row carries two synthetic columns:**

- `_local_id: string` — UUID v4 generated client-side at create time. Used as the primary key in Dexie AND as the URL slug (`/recommendations/<localId>`). On successful sync the server confirms the id is canonical (per §3.3) — the value never changes.
- `_sync_state: 'clean' | 'dirty_create' | 'dirty_update' | 'dirty_delete' | 'conflicting'` — the local sync state machine. `clean` = matches the server; `dirty_*` = pending operation in `sync_queue`; `conflicting` = a 409 from the server is awaiting rep resolution in `sync_conflicts`.

**Read-only caches** (`inspection_templates`, `legal_clauses`) do NOT carry `_local_id` or `_sync_state` — they mirror the server's canonical id and refresh on each successful sync (§3.9). The corpus is small enough (10s of clauses for OHSA + CLC Part II, ~50–100KB compressed) to ship in IndexedDB; offline citation insertion is therefore possible without the picker hitting the network.

**On-disk encryption posture.** IndexedDB is NOT encrypted at rest on the device. The rep's phone may be lost, stolen, or seized; the bytes on disk are recoverable by anyone with physical access and a forensic tool. The Dexie store therefore carries:

- **Envelope-encrypted ciphertext for every sensitive field** (`title_ct`, `body_ct`, `description_ct`, `observation_ct`, etc.). The workplace public key ships to the browser via `/api/auth/session` (already in place per `apps/api/src/routes/auth/session.ts:106`); the browser seals fresh writes under that key before they hit Dexie or the queue. The workplace _private_ key never leaves the server, so a stolen phone yields only ciphertext.
- **Plaintext for non-PI metadata only** (status enums, timestamps, ids, section names, jurisdiction codes, zone ids, citation markers like `[[cite:N]]`, recommendation numbers, action-item types). These were never encrypted in Postgres either (`docs/adr/0008-recommendations.md` §3.2 list projection); the local cache mirrors the server's projection shape.
- **NEVER plaintext decrypt results.** When a rep opens a `reveal` screen, the decrypted plaintext lives in JS heap memory for the lifetime of the view component and is dropped when the rep navigates away. It is NOT persisted to Dexie; the reveal route is always live-fetched from the server. This is the documented tradeoff — convenience vs the stolen-phone bound. The runbook calls it out.
- **NEVER step-up grants.** The auth cookie is `SameSite=Strict` + `HttpOnly` and never visible to JS, let alone persisted to Dexie. Step-up freshness state lives only in the session cookie's claims; it cannot be cached offline (see §3.6).

A future 1.12 schema change for IndexedDB-at-rest encryption (deferred per §"Don't") would be a version 2 bump that copies rows into an encrypted shape; out of scope for 1.10.

### 3.2 Sync queue model (`SyncOperation` row shape + attempts + backoff + dead-letter)

Every mutation that today flows through the typed API client (`apps/web/src/hazards/api.ts`, `action-items/api.ts`, `inspections/api.ts`, `recommendations/api.ts`, `evidence/api.ts`) writes BOTH to the local Dexie table optimistically AND appends a `SyncOperation` row to `sync_queue` with shape `{id, kind: 'create'|'update'|'delete'|'transition'|'evidence_upload', table, entity_local_id, payload_ct, payload_dek_ct, payload_hash, http_method, endpoint, if_match_etag, attempt_count, next_attempt_at, last_error, last_attempted_at, created_at, dead_letter}`.

**The payload is encrypted at queue-write time** using the workplace public key from `/api/auth/session`. The encryption is the same envelope shape as evidence DEKs in 1.7 (sealed-box around a per-payload DEK; `(payload_ct, payload_dek_ct)` mirror the server-side `(*_ct, *_dek_ct)` pair). The plaintext lives in JS memory only for the encrypt call; it is never persisted in Dexie. This is the structural defense against the stolen-phone bound: a forensic dump of IndexedDB yields ciphertext blobs whose DEKs are sealed to a key the rep doesn't have. The `payload_hash = sha256(canonical_json(payload))` is the idempotency key (§3.4).

**The queue worker** (`apps/web/src/sync/worker.ts`) runs in the main thread (no Web Worker; Dexie does not transfer well across the structured-clone boundary). It is invoked on app load (drain prior-session operations), network reachability change (`navigator.onLine` + a `/api/health` probe), every fresh enqueue, a 30-second poll while online + non-empty queue, and pull-to-refresh / manual "sync now". Hot loop: pull a batch of 10 rows where `next_attempt_at <= now()` and `!dead_letter`; for each, ship the request via the typed API client with the `Idempotency-Key: <payload_hash>` header (the queue does NOT decrypt the payload at drain — the ciphertext + DEK are passed straight to the API, which decrypts inside the request lifetime; the client's plaintext lifetime is just the encrypt call).

Per-response behavior:

- **2xx:** entity row's `_sync_state → 'clean'`; the server-returned canonical id is already the `_local_id` (per §3.3, no rewrite). Delete the queue row.
- **409 with conflict body:** write a `sync_conflicts` row (§3.7); mark entity `_sync_state = 'conflicting'`; delete the queue row (parked until the rep resolves).
- **400 / 422 (schema reject):** mark `dead_letter = true`. Not auto-retried — retrying produces the same rejection. Rep sees the error in the sync-status chip and decides delete-local vs fix-and-resubmit.
- **401 step-up:** should never happen (no queueable operation is step-up gated per §3.6). Defensive: mark `dead_letter = true`.
- **5xx / network failure:** increment `attempt_count`, bump `next_attempt_at` per the backoff curve, leave the queue row in place.

**Backoff curve:** `[1s, 5s, 30s, 5min, 30min, 2h, 12h, 24h]`. The `attempt_count` indexes this array; when `attempt_count >= 8` the row is marked `dead_letter = true`. Total time from first attempt to dead-letter is ~48h. The curve is a pure function (`apps/web/src/sync/backoff.ts: nextAttemptDelayMs(attemptCount)`) so it's unit-testable and the runbook can document it without copy-drift.

**Per-operation chain-anchor latency.** This is the load-bearing semantic: a queue operation queued at 10:14am that drains at 02:47am produces a chain row with `idx` allocated at 02:47am and `ts_ms` = 02:47am. The local Dexie row is rep-trusted before the drain; the chain anchor is the post-sync evidentiary anchor. The runbook documents this for arbitration purposes — a rep can produce queue-typed-at via their device clock + the local `created_at` column on the queue row, but the chain proves server-receipt time only. Non-negotiable #2 is preserved by this distinction; the chain is the canonical evidentiary surface, the queue is the recovery surface.

### 3.3 Optimistic ID strategy (client-provided UUID via `clientId` body field)

The client generates `_local_id` as a UUID v4 at create time. The local Dexie row uses it as the primary key; the URL uses it as the slug (`/recommendations/<localId>`); the sync queue references it as `entity_local_id`. **On the wire, every create POST carries a `clientId: string` field in the body, and the server INSERTs the row with `id = clientId` instead of `gen_random_uuid()`.**

This is option (b) from the scope brief — the simpler one. The alternatives were:

- **(a) Keep both `_local_id` and `server_id`, rewrite all FK references in local tables on first successful sync.** Two id columns per row, an entire rewrite phase on sync-success, the URL silently swaps mid-session. The complexity is in the wrong place: every consumer of the local row has to handle two ids and a moment of swap.
- **(b) Client-provided UUID, server INSERTs with it.** One id end-to-end. The URL is stable from the moment the rep types until the chain anchor lands. The chain payload's `recommendationId` / `hazardId` / etc. is the same uuid the rep saw in the URL bar. Foreign-key references (`recommendation_action_item_links.recommendation_id`, `recommendation_citations.recommendation_id`) work without rewrite.

Every existing create handler accepts a server-generated UUID today (`gen_random_uuid()` default in the migrations). The 1.10 ratchet is small: each handler accepts an optional `clientId: z.string().uuid()` body field, validates it does not collide with an existing row (returns 409 `id_collision` if so — the collision rate is negligible for v4 UUIDs but the check is cheap), and uses it as the insert id. The Zod schema's `.optional()` keeps backwards compatibility — an online client that does not generate a `clientId` continues to work; the server generates a UUID as before.

**The `client_provided_id` UNIQUE constraint already exists implicitly via the PK.** No new column, no new index. The chain anchor binds the same id end-to-end; the chain payload's `recommendationId` is the canonical id the moment the queue drains.

**Routes that take the ratchet:**

| Route                                     | Body field        | Notes                                  |
| ----------------------------------------- | ----------------- | -------------------------------------- |
| `POST /api/hazards`                       | `clientId?: uuid` | Used as `hazards.id`.                  |
| `POST /api/action-items`                  | `clientId?: uuid` | Used as `action_items.id`.             |
| `POST /api/action-items/:id/moves`        | `clientId?: uuid` | Used as `action_item_moves.id`.        |
| `POST /api/inspections`                   | `clientId?: uuid` | Used as `inspections.id`.              |
| `POST /api/inspections/:id/findings`      | `clientId?: uuid` | Used as `inspection_findings.id`.      |
| `POST /api/inspections/:id/signatures`    | `clientId?: uuid` | Used as `inspection_signatures.id`.    |
| `POST /api/recommendations`               | `clientId?: uuid` | Used as `recommendations.id`.          |
| `POST /api/recommendations/:id/responses` | `clientId?: uuid` | Used as `recommendation_responses.id`. |
| `POST /api/evidence` (finalize)           | `clientId?: uuid` | Used as `evidence_files.id`.           |

The transition / promote / submit / resolve / withdraw routes do NOT take a `clientId` — they don't create new top-level rows (the action-item bridge on `recommendation.submit` is a server-side transaction; the client doesn't need to predict that action_item's id because it never references it in a Dexie FK before the sync lands).

### 3.4 Idempotency model (`sync_idempotency` table + `Idempotency-Key` middleware + 7-day TTL)

The queue retries on 5xx / network failure. Without idempotency, a retry that drained but failed to return the response to the client (network blip on the response leg) would double-write — a second hazard, a second response row at `position=2`, a second action-item move. The mitigation is a deterministic client-side key on every mutation + a server-side dedup table.

**New `Idempotency-Key` middleware** (`apps/api/src/middleware/idempotency.ts`), wired into the root app BEFORE auth, body, and rate-limit so we can de-duplicate without consuming auth budget. Order in `apps/api/src/index.ts`: `csrfHeaderGuard()` → `idempotencyKeyGuard()` (NEW, short-circuits on replay BEFORE auth) → `authMiddleware()` → `rateLimit()` → route handlers.

The middleware contract: GET/HEAD → no-op; missing header → no-op (online clients without sync don't carry it); present → look up `sync_idempotency WHERE key = $1 AND user_id = $2` (scoped to actor — a different rep with the same key is a different operation). **On hit:** return the cached `response_status_code` + `response_body` verbatim. Do NOT re-run the handler. The chain anchor was emitted on the first run; re-emitting would produce a duplicate chain row for what is evidentially the same operation. **On miss:** call `next()`; after a 2xx, INSERT `{key, user_id, action_kind, entity_local_id, payload_hash, response_status_code, response_body, expires_at = now() + 7d}`. On 4xx/5xx, do NOT cache (the rep may retry with a corrected payload; caching a 422 would lock them into the failure).

**Table shape:** `sync_idempotency (key text primary key, user_id uuid not null references users(id) on delete restrict, action_kind text not null, entity_local_id uuid, payload_hash text not null, response_status_code smallint not null, response_body jsonb not null, created_at timestamptz not null default now(), expires_at timestamptz not null)`, with an index on `expires_at` for the sweep.

**The `payload_hash` cross-check defends against key collision.** If a rep replays the same key with a different body (extremely unlikely — the queue's `payload_hash` IS the key), the middleware returns 409 `idempotency_key_payload_mismatch` instead of the cached response. Defense against a future bug in the queue's key derivation.

**Existing chain-anchor semantics are preserved.** The 1.7 `evidence.uploaded`, 1.6 `action_item.moved`, 1.9 `recommendation.submitted` anchors — all are emitted inside the route handler on the FIRST successful drain. A retry that hits the idempotency cache returns the cached response (the existing row's id + ts) WITHOUT re-emitting. The chain row's `idx` is allocated once; the runbook documents this — "the chain proves the server saw this operation once, not the number of times the client retried." This is the critical interaction with the 1.7 `evidence.uploaded` chain anchor that the scope brief calls out: a finalize retry after a network blip returns the cached response (existing row's id + uploaded_at); the chain anchor binds to the first successful insert, not the retry.

**7-day TTL.** A nightly sweep job (`scripts/sync-idempotency-sweep.ts`, runs via `pg-boss`) deletes rows where `expires_at < now()`. A rep whose phone was offline for >7 days replays an operation that the server has forgotten, the handler runs again, the chain anchor fires again — acceptable because the dead-letter ceiling (§3.2) is 48h, so no operation survives the TTL window in practice. The TTL automation lands in 1.12; in 1.10 the table grows monotonically (acceptable at single-tenant scale).

### 3.5 Optimistic UI + reads-from-Dexie-first

Every list and detail view reads from Dexie first, renders immediately, then fires a background `fetch` to refresh. The typed API clients are wrapped so `listRecommendations()` (and siblings) returns `db.recommendations.where('_sync_state').notEqual('dirty_delete').toArray()` instantly, then fires `backgroundRefresh('/api/recommendations', reconcile)` without blocking the return. The reconcile pass merges per-row: server rows overwrite local `clean` rows; local `dirty_*` rows are left alone (the pending operation is still queued and will eventually drain); deleted-on-server rows that are still locally clean are deleted locally.

**Background refresh fires on:** view mount; window focus (`window.addEventListener('focus', ...)`); after every successful sync-queue drain (§3.2); pull-to-refresh on mobile + manual sync button on desktop (§3.11).

**Optimistic write surface.** Every mutation: (1) generate `_local_id` (uuid v4) for creates; (2) write to Dexie with the appropriate `_sync_state`; (3) append a row to `sync_queue` with the encrypted payload; (4) return the optimistic local row to the caller; (5) the queue worker drains in the background. The UI shows the optimistic state immediately. The sync-status chip (§3.11) shows the count of non-`clean` rows.

### 3.6 Queueable vs require-online classification

The architect's hard call: which writes are safe to queue offline, and which require online because they exercise a server-side capability that has no local equivalent. The decision is structural, not policy — it follows the data flow: anything that needs the workplace KEK, anything that needs step-up freshness, anything that produces a server-rendered artifact, anything that touches the auth lifecycle, requires online.

**Queueable** (drained offline → server when network returns):

The classification is enforced by the service worker's interception layer (§3.8). A queueable route offline returns 202 + the optimistic local state; a require-online route offline returns a synthetic 503 `network_required` so the UI renders the banner.

**Special case: `POST /api/recommendations/:id/submit`.** The submit is a server-side transaction (creates the linked action_item + emits `recommendation.submitted` chain anchor + allocates the per-jurisdiction `recommendation_number`). All three legs can land at drain time:

- The `recommendation_number` is allocated under an advisory lock at drain time (per ADR-0008 §3.2's `pg_advisory_xact_lock` pattern); the client never predicts it. The local optimistic view shows "Recommendation #pending" until the drain completes.
- The action_item creation uses the client-provided `clientId` for the recommendation but the action_item id is server-generated (we don't predict it). The recommendation_action_item_links row joins on the canonical ids at drain time.
- The chain anchor binds to server `submitted_at = now()` — NOT to the rep's queue-typed-at.

**This is the 21-day s.9(21) clock semantics call.** A recommendation submitted offline at 10:14am on Monday, sync'd at 02:47am on Tuesday, has its 21-day clock measured from 02:47am Tuesday — NOT from 10:14am Monday. The legal reasoning: OHSA s.9(21) says the employer must respond "within twenty-one days," and the legally relevant start point is when the employer learned of the recommendation, which is when the rep delivered it. Since the app's `submitted` status means "delivered to the workplace's employer co-chair" (per ADR-0008 §3.1 — _"there is no automated transport to MLITSD, ESDC, or any regulator"_), and the rep can only deliver after the queue drains and they actually hand the PDF to management, the chain anchor's `submitted_at` is the correct legal start. The rep needs to know this — a 12-hour offline-to-drain gap shifts the deadline by 12 hours. The UI surfaces it: when a rep submits offline, the optimistic view shows "Recommendation submitted (pending sync). 21-day clock starts on sync." The runbook documents the operator-side framing. See §3.12 for the full UX treatment.

### 3.7 Conflict resolution model (server-side `If-Match` etag + 409 + three-way merge for non-chain-anchored fields)

Every entity table that supports updates (hazards, action_items, inspections, recommendations, inspection_findings, recommendation_citations) gains a server-side `version` integer column, incremented on every PATCH. The column is projected to the client as the `etag` (a base64-encoded version) and the client sends `If-Match: <etag>` on every PATCH. Server-side: `SELECT version FOR UPDATE`; if row's version != etag, return 409 `{error: 'version_conflict', server_version, server_state, base_version}`; else `UPDATE SET ..., version = version + 1`, emit chain anchor, return 200 with new version.

**The client's queue worker on 409:** read the 409 body's `server_state`; read the local row's current state; read the Dexie row's `_base_state` (a new column on every updatable entity, the snapshot of the row at last `clean`); insert a `sync_conflicts` row with `{id, table, entity_local_id, kind, local_state_ct, server_state_ct, base_state_ct, surfaced_at}` (all three states envelope-encrypted under the workplace public key — same posture as queue payloads §3.2; conflict UI requires step-up + reveal to compare plaintext, per §3.8); mark entity `_sync_state = 'conflicting'`; surface via the amber sync-status chip.

**Three-way merge UI** (`apps/web/src/sync/conflict-resolve-view.tsx`): for non-encrypted fields (status, jurisdiction, section, recommendation_number, position numbers), shows three columns side-by-side with field-level "keep mine | keep theirs | merge" buttons. For encrypted text fields (hazard description, action_item description, recommendation body, inspection finding observation), shows the same three-column layout but each column requires a step-up reveal click — the rep is comparing two privileged versions of the same encrypted text, so the existing reveal endpoints fire for both. After the merge decision lands, the rep taps "Submit resolution" and the resolution becomes a fresh PATCH with `If-Match: <server's current version>` — which should succeed unless another conflict landed in the interim (rare; single-co-chair scope).

**Conflict-free operations** (no merge UI needed, no `If-Match` etag required):

- Inspection findings creation (`POST /api/inspections/:id/findings`) — append-only, each finding has a unique `(inspection_id, item_number)` pair. Two offline reps adding findings to the same inspection produce two appends; no conflict.
- Action_item creation (`POST /api/action-items`) — append-only by id.
- Action_item moves (`POST /api/action-items/:id/moves`) — append-only history. Two offline reps moving the same item to different sections produce two move rows; the latest move wins for the `action_items.section` column (server resolves at drain time by `moved_at DESC` — the chain captures both moves). This is `action_item_moves`-table semantics matching the ADR-0005 design (one row per event, each chain-anchored).
- Recommendation drafting (`POST /api/recommendations`) — append-only by id.
- Recommendation responses (`POST /api/recommendations/:id/responses`) — append-only by position; the queue worker handles position allocation by reserving `position = MAX + 1` at drain time under the advisory lock pattern (per T-R10 from ADR-0008's threat model).

**Chain-anchored event conflicts MUST surface to manual review.** Chain anchors are append-only — "keep mine" cannot overwrite the chain row. The conflict UI for these events frames it explicitly:

- `recommendation.submitted` — if a rep submits offline and a second submit lands online for the same recommendation (extremely rare; single-co-chair scope makes this near-zero, but theoretically possible if the rep has two devices), both anchors are in the chain. The UI surfaces: _"You submitted recommendation #3 offline at 10:14am; the server received another submit for #3 at 11:02am. Both submissions are now anchored in the audit chain. Pick which version of the recommendation text is canonical; the other is recorded as a withdrawn duplicate."_
- `recommendation.exported` / `inspection.exported` — exports require online (per §3.6); no conflict possible.
- `recommendation.resolved` / `recommendation.withdrawn` — terminal transitions. The conflict UI frames it as: _"You resolved this recommendation offline; the server received a withdrawal for the same recommendation. Both are anchored. The chain reflects the order: withdrawal at 11:02am, resolution attempt at 02:47am the next morning. Resolution failed (withdrawn is terminal); the chain row records the failure."_
- `inspection.signed` — append-only via `(inspection_id, signature_role)` unique. A second offline-signed signature for the same role lands as a 409 with the existing signature; the rep sees both and picks which one the audit chain reflects (typically the offline one was a draft; the server one is canonical — same posture as the recommendation submit conflict).
- `action_item.moved` — both moves anchor; the latest wins for the current section. No conflict UI needed (per "Conflict-free operations" above).

### 3.8 Encrypted-field conflict semantics + offline-evidence-upload flow

**Encrypted-field conflict UI requires step-up + online.** When the rep opens a conflict involving encrypted text (hazard description, recommendation body, finding observation), the conflict view fires three reveal calls — one for the base state, one for the local state (which is encrypted in Dexie under the workplace public key, so the SERVER decrypts the local ciphertext too — the rep's offline-typed body is sealed for the server's eyes), one for the server state. The reveal endpoint is `GET /api/recommendations/:id/reveal` (already in place per ADR-0008 §3.7's reveal route); a new sibling `POST /api/sync-conflicts/:id/reveal` accepts a `{which: 'local' | 'server' | 'base'}` parameter and decrypts the appropriate ciphertext from the `sync_conflicts` row, with the same step-up gate + chain anchor (`sync_conflict.revealed` kind, new in 1.10).

The queue's `_local_ciphertext` (the `payload_ct` field of a `dirty_update` operation, copied into `sync_conflicts.local_state_ct` on 409) is the snapshot at type-time — what the rep typed offline, sealed to the workplace public key. The server's ciphertext is the canonical current row. Comparison happens server-side after both decrypt; the diff is the plaintext-vs-plaintext diff that the conflict UI renders.

**Why the local ciphertext can be decrypted server-side.** Both the local-sealed payload AND the server-sealed canonical row are encrypted to the SAME workplace public key (server holds the private). The rep's local copy is not encrypted to a different key — it's encrypted to the same recipient. This is the structural payoff of the 1.7 sealed-box pattern: the rep's offline typing is sealed for the server's eyes, not for the rep's local consumption; the local Dexie row is "sealed write-only" from the rep's perspective.

**Offline-evidence-upload flow.** Evidence captures are the most complex offline operation because they're a three-step transaction across two origins (the Hono API + Tigris). The flow:

1. **Capture** — the rep taps capture-to-record; the browser opens camera; the photo is taken; the browser envelope-encrypts the bytes under the workplace public key (per ADR-0006); the resulting `_local_ciphertext_b64` is stored on the evidence_files Dexie row with `_local_storage_pending = true` AND a `sync_queue` operation of kind `'evidence_upload'` is enqueued. The local row's `_sync_state = 'dirty_create'`.
2. **Drain step (a) — request presigned URL.** Queue worker calls `POST /api/evidence/upload-url` with the encrypted ciphertext's SHA-256 + byte_size. Server returns the presigned PUT URL.
3. **Drain step (b) — PUT ciphertext to Tigris.** Queue worker does a foreground `fetch(presignedUrl, { method: 'PUT', body: ciphertextBlob })`. Service worker does NOT intercept this — Tigris is a different origin (§3.9 — the service worker only intercepts same-origin requests). On 5xx / network failure, the queue worker bumps `next_attempt_at` per the backoff curve. On success, the next step fires.
4. **Drain step (c) — finalize.** Queue worker calls `POST /api/evidence` with `{linked_type, linked_id, storage_key, sha256, byte_size, sealed_dek, clientId}` (the `clientId` is the `_local_id`). Server inserts the `evidence_files` row + emits `evidence.uploaded` chain anchor. On success, the Dexie row's `_local_ciphertext_b64` is deleted (the ciphertext is now on Tigris; keeping a local copy is unnecessary disk pressure) and `_sync_state = 'clean'`.

Each step has its own idempotency-key (the same `payload_hash` for steps (a) and (c); step (b) is naturally idempotent at the Tigris PUT layer because the storage_key is content-addressed by the SHA-256). A failure at any step rewinds: step (c) failure leaves the ciphertext on Tigris (orphan; the 1.7 sweep cleans it up via the existing nightly job); step (b) failure means we retry the PUT with the same presign URL until it expires (presigns are 5-minute TTL — the queue must retry within that window or re-request); step (a) failure means full retry from scratch.

**The single-queue-row pattern is preserved** — all three steps are one `sync_queue` row from the UI's perspective; the queue worker carries an internal `step` counter (0 = need presign, 1 = need PUT, 2 = need finalize) so a failure mid-flow resumes at the right step rather than starting over.

### 3.9 Service worker scope + caching strategy

The service worker is generated by `vite-plugin-pwa` (already in the locked stack per CLAUDE.md). The generated `sw.js` registers on first app load; subsequent navigations are served from the cache + the SW intercepts `/api/*` mutations for queueing.

**What the SW caches (all stale-while-revalidate unless noted):**

- **App shell** (HTML + JS + CSS bundles, build-hash addressed; immutable per build). On new bundle the SW emits `controllerchange` and the UI renders a discreet "Update available" toast.
- **Fonts** (Lucide icons, Source Serif 4, Inter, JetBrains Mono) — cache-first; immutable per build hash.
- **`GET /api/auth/session`** — per-session TTL; refresh on each successful sync drain. Returning reps see their workplace public key + signing key + display name instantly.
- **`GET /api/legal/clauses`** (legal corpus snapshot) — 24h TTL or per-session, whichever sooner. A re-seed produces a new `corpus_version`; the SW catches it on background-refresh. Offline citation insertion uses the cached snapshot — every citation the rep inserts offline pins to the cached corpus state; the server re-validates the triple at submit time and rejects retired versions per ADR-0008 §3.3.
- **`GET /api/inspection-templates`** (active versions) — refresh on sync drain.

**What the SW does NOT cache:** Tigris ciphertext (`GET /api/evidence/:id/decrypt` response bodies are session-bound + step-up gated; live only for the request); decrypted plaintext from any reveal endpoint; step-up grants (live in the auth cookie's claims); auth cookie itself (`HttpOnly` — SW can't see it).

**`/api/*` mutations:** NOT cached. The SW intercepts and queues (next section). **`/api/*` reads:** network-first with offline fallback to Dexie (handled via the typed API client's read-Dexie-first pattern per §3.5).

**Service worker scope is the app origin only.** Tigris uploads (cross-origin PUT to the presigned URL) are NOT intercepted; the queue worker handles them in the foreground (per §3.8 step b).

### 3.10 PWA install prompt + iOS 17 Add-to-Home-Screen

**Android Chrome + desktop Chrome:** standard `beforeinstallprompt` capture. The event handler stashes the prompt event in a React context; the app shell's top bar surfaces an "Install" button on small screens after the trigger criteria fire (≥3 sessions OR ≥1 piece of evidence captured — see below).

**iOS Safari 17+:** no programmatic install API. The "Install" button on iOS opens a modal with screenshots showing the share sheet → "Add to Home Screen" flow. The modal copy reads: _"On iPhone, JHSC Worker Hub installs via the share sheet. Tap the share icon, then 'Add to Home Screen.' Once installed, the app works offline."_ This is a deliberate restraint — per CLAUDE.md non-negotiable #10 (no marketing aesthetics), the modal is text + a single annotated screenshot, not a marketing flow.

**Prompt trigger criteria.** The install prompt fires from the app shell's top bar after either:

- The rep has used the app for ≥3 sessions (counted via `localStorage.sessionCount`, incremented on each first-page-load-after-auth), OR
- The rep has captured ≥1 piece of evidence (the offline-first value prop is most visible the first time the camera fires + the photo lands in the queue).

Prompt early but not on first session — the rep needs to see the app work before they install it. The trigger logic lives in `apps/web/src/pwa/install-prompt.ts`; the threshold is configurable per deploy via env (`WORKPLACE_PWA_INSTALL_PROMPT_SESSIONS = 3`).

**No programmatic install on iOS** is structural; we do not pretend otherwise. The runbook documents the iOS Safari path explicitly so a rep onboarding on an iPhone is not confused.

### 3.11 Sync status surface (app shell chip + pull-to-refresh + manual drain + dead-letter UI)

The sync-status chip lives in the top bar of the app shell (`apps/web/src/components/app-shell/top-bar.tsx`). It reads from a Zustand store fed by Dexie change events (Dexie's built-in `db.on('changes')` — no Dexie-observable dep per the §"Don't" guardrail) + `navigator.onLine` + a 30-second `/api/health` probe. Four states:

| State   | Color                                                | Copy                        | Trigger                                                                                |
| ------- | ---------------------------------------------------- | --------------------------- | -------------------------------------------------------------------------------------- |
| Synced  | Green (zinc with green dot for restrained aesthetic) | "Synced"                    | No `dirty_*` rows; no `conflicting` rows; online.                                      |
| Syncing | Blue                                                 | "Syncing N changes…"        | Non-zero `dirty_*` rows; online; queue worker actively draining.                       |
| Offline | Zinc                                                 | "Offline. N pending."       | `navigator.onLine = false` OR `/api/health` probe failed; non-zero dirty rows.         |
| Paused  | Amber                                                | "Sync paused. M conflicts." | Non-zero `conflicting` rows OR non-zero `dead_letter` rows. Tap to open conflict view. |

Tapping the chip opens `apps/web/src/sync/sync-status-view.tsx` — a list of pending operations + conflicts + dead-letter rows, with per-row actions (resolve conflict, retry dead-letter, delete local row). The view is mobile-primary (full-screen card list, sticky bottom action) per non-negotiable #9 + #10.

**Pull-to-refresh on mobile** (`apps/web/src/lib/pull-to-refresh.ts`) — touch gesture wrapped in a Framer Motion `useDragControls`. On release past threshold, fires the queue drain + the background-refresh fan-out (every list view's reconcile call). Haptic feedback on threshold (per CLAUDE.md mobile patterns).

**Manual drain button on desktop** — keyboard shortcut `⌘⇧S` opens the sync-status view + drains. The desktop sync chip in the sidebar shows the same state.

**Dead-letter UI** — a dead-letter row is surfaced with the server's error code (e.g. `citation_marker_density_violation` from a recommendation submit) + a "Fix and resubmit" or "Delete local row" action. The rep is the authority on which to choose; the app does not auto-decide. The runbook documents the operator-side recovery for the common dead-letter cases.

### 3.12 21-day s.9(21) clock semantics under offline submit

The clock is server-side, measured from `submitted_at` (the value the server stamps when the queue drains), per ADR-0008 §3.6 + the `computeRecommendationDeadline` pure function in `packages/shared-types/src/index.ts:331`. 1.10 does NOT change the function; it changes the rep-facing copy around it.

**The UX call:** when a rep taps "submit" offline, the optimistic detail view shows:

> Recommendation submitted (pending sync).
> 21-day clock starts on sync.

When the queue drains, the view updates to:

> Recommendation submitted 02:47am on 2026-06-03.
> 21 days remaining. Response due by 2026-06-24.

**The legal reasoning** is documented in the runbook: OHSA s.9(21) measures the clock from when the employer learns of the recommendation. The app's `submitted` status means "delivered to the employer co-chair" (per ADR-0008 §3.1 — _no automated transport to MLITSD, ESDC, or any regulator_). The rep can only deliver after the queue drains; the server's `submitted_at` is the closest proxy to that delivery moment that the app can anchor. A rep who typed offline 5 days before sync still has a 21-day clock measured from sync time — they had not delivered to management during those 5 days, so the clock had not started.

**The rep needs to know this.** If they type offline thinking the clock has started, they may miscount the deadline. The UI copy is explicit; the runbook is explicit; the chain row's `submitted_at` is the canonical legal timestamp. CA-FED recommendations (CLC s.135(6) — "as soon as possible") are unaffected because they have no fixed clock; the badge reads "days since submission: N" measured from the server's `submitted_at`, which is the right semantic regardless of offline.

## Consequences

### Positive

- **Non-negotiable #9 (mobile-primary) is structurally honored.** A rep on a freezer floor or a dock with 2G LTE can capture hazards, log inspection findings, move action items, draft recommendations, and capture photo evidence — all offline — and the queue drains when the bar returns. The "patchy LTE" failure mode that 1.5–1.9 left as a forward seam is now closed.
- **The chain remains the canonical evidentiary surface.** Per non-negotiable #2, every queued operation produces a chain anchor at drain time with server-side `ts_ms`. The local Dexie row is rep-trusted before sync; the chain is the post-sync record. The semantic distinction is documented; arbitration uses the chain.
- **Idempotency is structural, not ad-hoc.** The `Idempotency-Key` middleware + `sync_idempotency` table + `payload_hash` cross-check means a retry storm produces zero duplicate writes; the chain rows are emitted once per logical operation. The 1.4–1.9 routes don't double-write on a network blip mid-finalize, and the cross-check defends against key collision.
- **Client-provided UUIDs simplify the entire end-to-end story.** The `_local_id` the rep types against is the same uuid the chain anchor binds to. No rewrite phase, no two-id-per-row dance, no URL silently swapping mid-session. The Zod `.optional()` shape preserves backwards compatibility with online clients that don't generate `clientId`.
- **The polymorphic-FK ratchet pattern (ADR-0007 §3.7 → ADR-0008 §3.5) is not extended.** The queue is plumbing, not provenance; no new `source_type` value, no new `linked_type` value. The existing enums absorb the offline writes cleanly.
- **The stolen-phone bound is the workplace public key + sealed-box envelope, not the IndexedDB layer.** A forensic dump of IndexedDB yields envelope-encrypted ciphertext whose DEKs are sealed to a key the rep doesn't have; the workplace private key lives on the server. This is the same posture as 1.7 evidence at-rest on Tigris; 1.10 extends it to the on-device cache.
- **The legal corpus is shipped offline.** Citation insertion works without network; the picker UI is the same online and offline; the corpus snapshot refreshes daily or per-session. The 21-day clock arithmetic in `computeRecommendationDeadline` is a pure function in `packages/shared-types` that runs identically on the client and server.

### Negative / accepted tradeoffs

- **Every existing 1.5–1.9 route takes a ratchet (clientId + If-Match + version column + idempotency-key honoring).** The S2 surface is wide; the integration tests live across every prior milestone's route file. The mitigation is the ratchet being mechanical: each route's diff is small and similar (Zod optional field; PATCH 409 path; SELECT `version`; UPDATE `version + 1`). The migration linter (existing per 1.6 sec-F1 shape) catches drift.
- **IndexedDB is not encrypted at rest.** A stolen phone yields ciphertext for sensitive fields, but plaintext for non-PI metadata (status enums, timestamps, ids, citation markers, recommendation numbers). The metadata leakage is the documented tradeoff — a rep's pattern of activity is visible to a forensic adversary even if the message content is not. 1.12 absorbs the structural fix (browser-native at-rest encryption via WebCrypto + Indexed-DB encryption wrapper is the long-term answer; deferred per §"Don't").
- **The chain-anchor latency is rep-facing and may confuse a casual user.** A recommendation typed offline at 10:14am that anchors at 02:47am the next morning has its evidentiary timestamp at 02:47am. The runbook documents this; the UI copy makes it explicit; the threat model treats it as a privacy + evidentiary surface. Arbitration adjudicators reading the chain need to know that "queue-typed-at" is a separate timeline the rep can produce from their device. The 1.10 chain anchor is server-receipt time, full stop.
- **Conflict resolution for encrypted fields requires online + step-up.** A rep with two devices who hits a conflict on a recommendation body cannot resolve it offline — they need to be online to fire the reveal endpoints on both ciphertexts. This is the same posture as every existing reveal flow; the cost is that conflicts on encrypted fields cannot be cleared during the offline session. The amber sync-status chip surfaces them; the rep resolves when they next reach network.
- **Idempotency-Key middleware runs before auth.** A malicious unauthenticated client could spam idempotency keys to probe the cache. Mitigations: the keys are random UUIDs (the queue's `payload_hash`); the cache is keyed on `(key, user_id)` so an unauthenticated client cannot hit any cached entry; the rate-limit middleware (post-auth) still bounds the request volume. Documented as a sec-F-class threat in the §S0 model.
- **The sync_idempotency table grows monotonically through 1.10.** The TTL sweep job is 1.12. At single-tenant scale (~1 co-chair + 5–10 reps writing a few dozen operations a day), the table grows by ~1000 rows/week — acceptable for 1.10. 1.12 closes the structural gap.
- **iOS Safari has no programmatic install.** The "Install" button on iOS opens an instructions modal rather than triggering the OS dialog. This is an iOS Safari limitation; we don't pretend otherwise. The runbook + the modal copy state it plainly.
- **The conflict UI for `recommendation.submitted` (the rarest case) is human-mediated.** Two devices submitting offline against the same draft can produce two chain anchors; the resolution is the rep choosing which is canonical and the other is recorded as a withdrawn duplicate. Single-co-chair scope makes this near-zero in practice; the runbook documents the recovery procedure for the pathological case.

### Risks

- **A bug in the `If-Match` etag path could let stale PATCHes succeed silently.** Mitigation: integration test covering `version` increment on every PATCH; the `version` column is non-nullable; a PATCH without `If-Match` returns 428 `precondition_required` (online or offline). The web client always sets `If-Match`; the queue worker captures it at type-time + sends it at drain time.
- **A bug in the queue's payload-hash derivation could produce idempotency-key collisions.** Mitigation: `payload_hash = sha256(canonical_json(payload))` is a pure function with a unit test; the `(key, user_id)` cache key plus the `payload_hash` cross-check are the two defenses. A collision returns 409 `idempotency_key_payload_mismatch` rather than the cached response.
- **A pathological offline duration past the 48h dead-letter ceiling means operations surface to manual review.** Mitigation: the dead-letter UI is explicit; the rep sees what didn't drain and decides; the runbook covers the common cases (network-failure storms, server outage, schema rejection).
- **The legal-corpus snapshot can be stale on a submit attempt.** A rep with a snapshot from yesterday tries to submit a recommendation against a clause that was retired today. Mitigation: the server's submit handler re-validates the triple against the live corpus at drain time (per ADR-0008 §3.3 — `citation_not_in_corpus` + the retired-version check); the dead-letter UI surfaces the rejection and the rep updates the citation. The corpus's append-only-version invariant (1.4 T-LC3 + T-LC5) means historical triples remain queryable, so a recommendation typed against a since-retired version still validates if the version is the one the rep pinned (not the live active version).
- **Service-worker scope misregistration could let a stale SW cache an out-of-date app shell indefinitely.** Mitigation: the `vite-plugin-pwa` defaults include `skipWaiting + clientsClaim` with a `controllerchange` UI toast; the SW is build-hash-addressed; the runbook covers force-refresh procedures.
- **A rep's device clock can be wrong, producing a wrong `_local_created_at` on Dexie rows.** The chain anchor's `ts_ms` is server-side and unaffected; the local clock skew shows up only on the dead-letter UI's "typed at" hint. Documented; not a server-side risk.

## Compliance check

- **#2 chain-of-custody.** Every queued operation produces a chain anchor at server-drain time with server-side `ts_ms`. The local Dexie row is rep-trusted; the chain is the post-sync evidentiary record. No queued mutation bypasses the chain. The chain-anchor latency is explicit in the rep-facing copy + the runbook.
- **#4 privacy-by-default + minimize collection.** Sensitive fields in Dexie are envelope-encrypted under the workplace public key (server holds the private). Plaintext lives in JS heap memory only for the lifetime of a reveal view; never persisted. Sync queue payloads are sealed. The 1.10 surface adds zero new plaintext-at-rest paths.
- **#9 mobile-primary.** The whole milestone is the mobile-primary closure of the 1.5–1.9 forward seams. Pull-to-refresh, sticky bottom actions, full-screen detail, touch-target ≥44pt sync-status chip, haptic feedback on drain — the entire UI surface respects the 390px phone-first posture.
- **#10 restrained legal-grade aesthetic.** The sync-status chip is text + dot, no marketing flourish; the conflict-resolution view is a three-column diff, no celebratory animations; the PWA install modal is text + a single annotated screenshot, no promotional copy. The iOS install modal explicitly avoids "Install JHSC Worker Hub now! Get the full mobile experience!" — the copy is documentary.
- **#11 Excel imports browser-side parse/encrypt.** 1.10 does not ship Excel imports (that's 1.11). The 1.10 schema is built to accommodate ≥1000 sync_queue rows in one drain pass (per the §"Critical guardrails" line) so 1.11 can hand large batches to the queue without UI lag — the queue's batch-of-10 read shape (§3.2) scales linearly with row count, not super-linearly.
- **#16 step-up + audit on every export.** Exports are require-online (per §3.6); offline export attempts surface the "Network required" banner. The step-up + chain-anchor + signed-PDF + Tigris discipline from ADR-0007 + ADR-0008 is unaffected.
